# Agent Council v0.2.0 — Event-Driven Optimization Design

**Date:** 2026-04-13
**Status:** Approved
**Release:** v0.2.0 (all features in single release)

## Summary

Five features inspired by oh-my-openagent (OMO), integrated via an event-driven architecture rewrite:

1. **Intent Gate** — classify human messages before processing
2. **Facilitator Agent** — LLM-powered discussion moderator
3. **Agent-Autonomous Model Selection** — per-agent model tiers with fallback
4. **Accumulated Learning** — proactive memory injection at session-start and per-turn
5. **Parallel Execution with Worktree** — dispatch workers to git worktrees post-deliberation

**Infrastructure change:** Event Bus Core — typed event system replacing monolithic router (enables all 5 features).

## Design Decisions (from brainstorming)

| Decision | Choice | Alternatives Rejected |
|----------|--------|-----------------------|
| Release strategy | All in v0.2.0 | Phased releases, partial spec |
| Parallel execution scope | Full: worktree + auto code generation | Lightweight framework only, sequential |
| Facilitator type | Full LLM agent | System logic only, hybrid |
| Model selection | Agent-autonomous (per-agent config) | Fixed degradation table, per-turn global override |
| Learning injection | Both layers (session-start + per-turn) | Session-start only, per-turn only |
| Architecture | Event-driven | Pipeline, augmented router |

## 1. Event Bus Core

**New file:** `src/events/bus.ts`

Typed EventEmitter wrapping Node.js `EventEmitter`. No external dependencies.

### Event Catalog

| Event | Payload | Emitter | Subscribers |
|-------|---------|---------|-------------|
| `message.received` | `{ message: CouncilMessage; threadId: number }` | GatewayRouter | IntentGate, ParticipationManager |
| `intent.classified` | `{ intent: IntentType; complexity: Complexity; threadId: number; message: CouncilMessage }` | IntentGate | DeliberationHandler, ActiveRecall |
| `deliberation.started` | `{ threadId: number; participants: string[]; roles: Record<string, AgentRole>; structure: DebateStructure }` | DeliberationHandler | FacilitatorAgent, ActiveRecall |
| `memory.injected` | `{ threadId: number; agentId: string; memories: string[] }` | ActiveRecall | (logging) |
| `agent.responding` | `{ threadId: number; agentId: string; role: AgentRole }` | DeliberationHandler | ActiveRecall |
| `agent.responded` | `{ threadId: number; agentId: string; response: ProviderResponse; role: AgentRole; classification: ResponseClassification }` | DeliberationHandler | AntiSycophancyHandler, PatternDetectionHandler, FacilitatorAgent, UsageTracker |
| `pattern.detected` | `{ threadId: number; pattern: PatternType; targetAgent: string }` | PatternDetectionHandler | FacilitatorAgent |
| `convergence.detected` | `{ threadId: number; angle: string }` | AntiSycophancyHandler | FacilitatorAgent |
| `facilitator.intervened` | `{ threadId: number; action: FacilitatorAction; content: string; targetAgent?: string }` | FacilitatorAgent | DeliberationHandler, GatewayRouter |
| `deliberation.ended` | `{ threadId: number; conclusion: string; intent: IntentType }` | DeliberationHandler / FacilitatorAgent | ExecutionDispatcher, SessionLifecycle |
| `execution.dispatched` | `{ threadId: number; tasks: ExecutionTask[] }` | ExecutionDispatcher | (logging) |
| `execution.completed` | `{ threadId: number; tasks: ExecutionTask[]; diffs: string[] }` | ExecutionDispatcher | ExecutionReviewer |
| `session.ending` | `{ threadId: number; trigger: 'keyword' \| 'timeout' \| 'max_turns' }` | GatewayRouter | SessionLifecycle |
| `session.ended` | `{ threadId: number; topic: string; outcome: string }` | SessionLifecycle | GatewayRouter (cleanup) |

### Implementation

```typescript
import { EventEmitter } from 'node:events';

interface EventMap {
  'message.received': { message: CouncilMessage; threadId: number };
  'intent.classified': { intent: IntentType; complexity: Complexity; threadId: number; message: CouncilMessage };
  // ... all events above
}

export class EventBus {
  private emitter = new EventEmitter();

  emit<K extends keyof EventMap>(event: K, payload: EventMap[K]): void {
    this.emitter.emit(event, payload);
  }

  on<K extends keyof EventMap>(event: K, handler: (payload: EventMap[K]) => void | Promise<void>): void {
    this.emitter.on(event, handler);
  }

  once<K extends keyof EventMap>(event: K, handler: (payload: EventMap[K]) => void | Promise<void>): void {
    this.emitter.once(event, handler);
  }

  off<K extends keyof EventMap>(event: K, handler: (payload: EventMap[K]) => void | Promise<void>): void {
    this.emitter.off(event, handler);
  }
}
```

### GatewayRouter Refactoring

Current router (332 lines) becomes a thin orchestrator (~80 lines):
- Manages `sessions` Map (session state per thread)
- Receives Telegram messages → emits `message.received`
- Subscribes to events that require Telegram sends → calls `sendFn`
- Manages inactivity timers
- Handles `session.ending` / `session.ended` cleanup

All deliberation logic moves to `src/council/deliberation.ts`.

## 2. Intent Gate

**New file:** `src/council/intent-gate.ts`

Subscribes to `message.received`, emits `intent.classified`.

### Intent Types

| Type | Description | Effect on Deliberation |
|------|-------------|----------------------|
| `deliberation` | Needs multi-agent debate | Normal rounds (config: 3) |
| `quick-answer` | Simple question | max_inter_agent_rounds = 1 |
| `implementation` | Code request | Normal debate, then triggers execution dispatch |
| `investigation` | Research/debug question | Same deliberation loop, but agents receive a research-first prompt injection: "Before debating, gather evidence and present findings." No separate mechanism. |
| `meta` | Council commands (end, topic change) | Handled directly, no deliberation |

### Complexity Levels

| Level | Description | Model Tier Used |
|-------|-------------|----------------|
| `low` | Trivial, factual | Agent's `models.low` |
| `medium` | Standard | Agent's `models.medium` (default) |
| `high` | Complex reasoning | Agent's `models.high` |

### Classification Strategy

1. **Keyword matching** (free, fast): Pattern-based rules attempt classification first.
   - `meta` keywords: end keywords from config + "換個角度", "重新開始"
   - `implementation` keywords: "實作", "implement", "寫", "build", "create", "add feature"
   - `investigation` keywords: "為什麼", "why", "debug", "調查", "怎麼壞的"
   - `quick-answer` keywords: single question mark + short message (< 50 chars)
   - `deliberation`: default fallback
2. **Confidence check**: If keyword confidence < 0.7, fallback to Haiku LLM call for classification.
3. **Complexity heuristic**: Based on message length, topic count, and presence of code blocks.

## 3. Facilitator Agent

**New file:** `src/council/facilitator.ts`
**New config:** `config/agents/facilitator.yaml`

### AgentConfig Extension

```typescript
interface AgentConfig {
  // ... existing fields
  roleType?: 'peer' | 'facilitator';  // default: 'peer'
}
```

### Facilitator Behavior

- **Is** an AgentWorker with its own LLM provider, model config, bot token, and Telegram identity.
- **Is not** assigned deliberation roles (advocate/critic/etc.) — always acts as facilitator.
- **Is not** targeted by anti-sycophancy challenge injection (not a debater).
- **Is** monitored by pattern detector (to prevent bias).

### FacilitatorAction Types

| Action | Trigger | Example Output |
|--------|---------|----------------|
| `steer` | Topic drift detected | "We're drifting — let's return to X." |
| `challenge` | Agent avoided a point | "Huahua, you didn't address Binbin's concern about Y." |
| `summarize` | After N rounds, or on request | "Current consensus: A. Disagreements: B, C." |
| `escalate` | Deadlock detected | "You disagree on X. Human, your call." |
| `structure` | At deliberation start | "Let's do structured debate: each state your position, then cross-examine." |
| `end` | Arguments exhausted | "Both sides are fully explored. Ready for conclusion." |

### Event Subscriptions

| Event | Action |
|-------|--------|
| `deliberation.started` | Announce structure, post opening message |
| `agent.responded` | Monitor response quality; decide if intervention needed |
| `convergence.detected` | Decide: allow convergence or force deeper exploration |
| `pattern.detected` | Issue targeted guidance to affected agent |

### Intervention Decision

Facilitator uses its LLM to decide whether to intervene after each `agent.responded`. System prompt includes:
- Current deliberation history
- Convergence state
- Detected patterns
- Instruction: "Intervene ONLY when it improves deliberation quality. Most turns need no intervention."

When intervening, emits `facilitator.intervened` with action type and content. DeliberationHandler inserts this as a message in the conversation.

## 4. Agent-Autonomous Model Selection

### AgentConfig Extension

```typescript
interface AgentConfig {
  // ... existing fields
  model: string;                        // kept for backwards compatibility
  models?: {
    low: string;
    medium: string;
    high: string;
  };
  defaultModelTier?: 'low' | 'medium' | 'high';  // default: 'medium'
}
```

**Backwards compatibility:** If `models` is not set, all tiers use the `model` field value.

### Model Resolution Flow

1. `intent.classified` carries `complexity: Complexity`.
2. `AgentWorker.respond()` receives complexity as parameter.
3. Worker resolves model: `this.config.models?.[complexity] ?? this.config.model`.
4. Passes resolved model to `provider.chat()`.

### Provider Fallback

Each provider class gains a `chatWithFallback()` method:

```typescript
async chatWithFallback(
  messages: ProviderMessage[],
  options: ChatOptions,
  fallbackModels: string[],
): Promise<ProviderResponse> {
  for (const model of [options.model, ...fallbackModels]) {
    try {
      return await this.chat(messages, { ...options, model });
    } catch (err) {
      if (isRetryableError(err) && model !== fallbackModels.at(-1)) {
        console.log(`[Fallback] ${model} failed, trying next...`);
        continue;
      }
      throw err;
    }
  }
  throw new Error('All models exhausted');
}
```

Retryable errors: HTTP 429 (rate limit), 503 (service unavailable), 529 (overloaded).

### Token Statistics Enhancement

```typescript
interface AgentStats {

    calls: number;
    inputTokens: number;
    outputTokens: number;
  }>;
}
```

## 5. Accumulated Learning

**New file:** `src/memory/active-recall.ts`

### Layer 1: Session-Start Injection

Subscribes to `deliberation.started`.

1. Extract keywords from the triggering human message.
2. Query `brain.db` via FTS5 for matching `principle` and `rule` type memories with `confidence >= 0.7`.
3. Build injection block:

```
== Relevant Historical Decisions ==
[ref:principle-architecture.md] Past tendency: favor monorepo with clear module boundaries.
[ref:session-2026-04-10-api-design.md] Last API design debate chose REST over GraphQL (confidence 0.8).

You may continue, challenge, or overturn these conclusions. State your stance explicitly.
```

4. Inject into each agent's system prompt via a new `ActiveRecall.getSessionContext(agentId, keywords)` method called by `buildSystemPrompt()`.

### Layer 2: Per-Turn Injection

Subscribes to `agent.responding` (before LLM call).

1. Analyze current turn messages (human + other agents' responses so far).
2. FTS5 query for memories relevant to the specific arguments in this turn.
3. Inject as additional challenge prompt content (appended after anti-sycophancy challenge).
4. **Deduplication:** Track `Set<memoryId>` per session. Same memory injected at most 2 times (once at session start, once per-turn).

### Decision Rule Extraction (Consolidator Enhancement)

**Modified file:** `src/memory/consolidator.ts`

Current consolidation extracts:
- `principle` (2-3 sentences)
- `pattern` (behavioral tendency)

**New output:**
- `decision_rules: string[]` — concrete rules in format: "When encountering X, prioritize Y"

Extraction conditions:
- Only from sessions with `outcome = 'decision'` and `confidence >= 0.7`

### MemoryRecord Type Extension

```typescript
type MemoryType = 'session' | 'principle' | 'rule' | 'archive';
```

New `rule` type:
- `confidence`: 0.85 (between session and principle)
- Stored as individual records in brain.db
- Searchable via FTS5

### DB Schema Change

```sql
-- No new table needed; rules use the existing memories table with type = 'rule'
-- FTS5 index already covers all memory types
```

## 6. Parallel Execution with Worktree

**New directory:** `src/execution/`

### Files

| File | Responsibility |
|------|---------------|
| `dispatcher.ts` | Subscribe to `deliberation.ended`; extract tasks; create worktrees; spawn executors |
| `worktree.ts` | Git worktree lifecycle (create, remove, list) |
| `executor.ts` | Run a single agent in an isolated worktree to generate code |
| `reviewer.ts` | Collect diffs from completed executors; trigger council code review |

### Trigger Flow

1. `deliberation.ended` fires with `intent = 'implementation'`.
2. Dispatcher uses LLM to decompose the deliberation conclusion into discrete tasks.
3. For each task: `git worktree add /tmp/council-{taskId}-{timestamp} -b council/{taskId}`.
4. Executor spawns an AgentWorker in the worktree context.
5. Executor writes code, commits to the branch.
6. On completion: emit `execution.completed` with diffs.
7. Reviewer presents diffs to peer agents for code review.
8. Facilitator summarizes review results.
9. Human decides which branches to merge.

### Worktree Manager

```typescript
interface WorktreeInfo {
  taskId: string;
  path: string;
  branch: string;
  createdAt: number;
}

export class WorktreeManager {
  private active: Map<string, WorktreeInfo> = new Map();
  private maxConcurrent: number;
  private repoPath: string;

  async create(taskId: string): Promise<WorktreeInfo>;
  async remove(taskId: string): Promise<void>;
  async removeAll(): Promise<void>;
  list(): WorktreeInfo[];
  isAtCapacity(): boolean;
}
```

Safety mechanisms:
- Max concurrent worktrees: 3 (configurable)
- Executor timeout: 300,000ms (5 minutes, configurable)
- Auto-cleanup on process exit (remove all worktrees)
- Branch naming: `council/{taskId}` for easy identification

### ExecutionTask Type

```typescript
interface ExecutionTask {
  id: string;
  description: string;
  assignedAgent: string;
  worktreePath: string;
  branch: string;
  status: 'pending' | 'running' | 'completed' | 'failed';
  result?: {
    diff: string;
    filesChanged: string[];
    commitHash: string;
  };
  error?: string;
}
```

### Config Extension (council.yaml)

```yaml

  enabled: true
  max_concurrent_worktrees: 3
  executor_timeout_ms: 300000
  auto_dispatch: true  # false = ask human before dispatching
  repo_path: .         # path to the git repository
```
