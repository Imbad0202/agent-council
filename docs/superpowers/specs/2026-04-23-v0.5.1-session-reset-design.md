# v0.5.1 Session Reset — Design

**Version target**: v0.5.1
**Theme**: Long-running council sessions (Anthropic *Harness Design for Long-Running Applications*, 2026-04)
**Roadmap anchor**: `agent-council-ROADMAP.md` §0.5.1
**Date**: 2026-04-23
**Supersedes**: 2026-04-23 initial draft (pre-Codex scope reduction)

---

## 0. Scope revision (2026-04-23, post-Codex review)

The initial v0.5.1 spec bundled four features:

- (A) Segment history + persist reset snapshots
- (B) Carry forward snapshot as cached prefix
- (C) `ModelRegistry` via Anthropic Models API `max_input_tokens`
- (D) 80% context-usage passive hint + cache-token instrumentation

An external cross-model review (Codex, 2026-04-23) surfaced three structural hazards:

1. **Multi-provider carry-forward broken.** `OpenAIProvider` / `GeminiProvider` ignore `systemPromptParts` entirely. A reset under a mixed-provider council would resume with Claude agents reading the summary and OpenAI / Gemini agents starting from an empty segment — the exact incoherence v0.5.1 is supposed to prevent.
2. **Cache-hit not guaranteed.** The summary bytes are not stabilised at source. Facilitator temperature defaults to 0.7 (or 1 under thinking) in `ClaudeProvider`. Exit criterion "`cache_read_input_tokens > 0` on turn 2" cannot be asserted.
3. **`BlindReviewStore.create()` rejects segment 1 start if segment 0 has an unrevealed session.** Deferring the interaction to v0.5.2 is not cosmetic — it is operational breakage the first time a user runs `/councilreset` mid blind-review.

Decision: v0.5.1 is re-scoped to **(A) + (B, provider-agnostic) + blind-review-reset interaction**. The cache-hit instrumentation, 80% hint, and `ModelRegistry` move to v0.5.2. Rationale:

- (A + B) is the user-visible primitive. Without it, v0.5.1 ships nothing.
- Provider-agnostic carry-forward was the unstated requirement. Skipping it yields a feature that works only for Claude-only councils.
- The blind-review-reset guard is a small code change (5–10 lines) that prevents a silent breakage class.
- `ModelRegistry` + 80% hint is a *secondary signal*. Users can see context growth themselves. Shipping a broken passive hint is worse than shipping no hint.
- Cache-hit can be instrumented later once (A + B) lands — the design is forward-compatible.

## 1. Problem

Council sessions in a single Telegram chat (or CLI session) accumulate an ever-growing message history. v0.4.0 prompt caching mitigates token cost but not semantic drift — past ~100 turns, agent responses start echoing earlier rounds and sessions "lose the plot". There is no user-facing primitive to mark a conceptual boundary ("we're done with topic A, starting topic B") while preserving audit.

## 2. Goal

Ship a user-triggered `/councilreset` command that:

1. Produces a structured session summary (decisions, open questions, evidence pointers, blind-review state) and persists it.
2. On next turn, feeds the summary to **every agent regardless of provider** via a provider-agnostic prepend. Prior turns remain readable via `/councilhistory` for audit but are not resent to agents.
3. Refuses to run while an unrevealed blind-review session is active on the same thread, and tells the user what to do instead.

**Non-goals** (explicit, enforced):

- Automatic compaction / auto-reset. Reset is always user-triggered.
- `ModelRegistry` lookup via Anthropic Models API. Deferred to v0.5.2.
- 80% context-usage passive hint. Deferred to v0.5.2.
- `cache_creation_input_tokens` / `cache_read_input_tokens` instrumentation. Deferred to v0.5.2 (needed to *verify* cache-hit; v0.5.1 ships the feature without that verification metric).
- Per-agent snapshot variants. All agents share one snapshot per reset.
- Cross-chat / cross-session snapshot reuse. Deferred to v0.5.3.
- `/councilcontract` sprint contract. Deferred to v0.5.2.
- `--carry-forward-blind-review-only` flag. Deferred to v0.5.3.
- Durable segment messages across restart. Deferred to v0.5.2.

## 4. Components

### 4.1 `SessionReset` (`src/council/session-reset.ts`)

Takes `(db, facilitator)` and operates on `(handler, threadId)`:

1. **Guard**: if `handler.getBlindReviewSessionId(threadId) !== null`, throw `BlindReviewActiveError` with user-facing message "A blind-review session is pending. Reveal it via `/blindreview reveal` or cancel via `/cancelreview` before resetting." No facilitator call, no tokens burned.
2. Call facilitator via a new `AgentWorker.respondDeterministic(...)` method (adds `temperature: 0` override). Messages = current segment + synthetic reset-summary prompt. Explicit `extraSystemParts = undefined` (snapshot from any prior segment MUST NOT enter this call — see Codex finding #2).
3. Parse minimal metadata (counts of `## Decisions` / `## Open Questions` bullet lines; `blindReviewSessionId` is `null` because of guard at step 1; `currentTopic` from `handler.getCurrentTopic(threadId)`).
4. Persist snapshot via `ResetSnapshotDB.recordSnapshot()`.
5. `handler.sealCurrentSegment(threadId, snapshotId)` + `handler.openNewSegment(threadId)`, wrapped in `try / catch`: if either throws, delete the snapshot (`db.deleteSnapshot(snapshotId)`) so `/councilhistory` does not show a sealed segment the live session never adopted. Re-throw.

Returns `{ snapshotId, summaryMarkdown, metadata, segmentIndex }`.

**Failure modes**:

- Blind-review active → abort pre-facilitator (no DB write, no mutation).
- Facilitator call fails → abort post-guard (no DB write, no mutation).
- DB write fails → abort post-facilitator (facilitator tokens burned, no mutation; user can retry).
- Seal / open fails → rollback: snapshot deleted from DB, segment state untouched.

### 4.2 `ResetSnapshotDB` (`src/storage/reset-snapshot-db.ts`)

Pattern-matches `BlindReviewDB`. Schema in §5. Transaction boundary: `recordSnapshot()` is a single `INSERT`, atomic. `deleteSnapshot()` is a single `DELETE`, atomic. No multi-row transactions required in v0.5.1.

### 4.3 Provider-agnostic snapshot prepend (`src/worker/agent-worker.ts`)

Current `AgentWorker.respond()` flow:

```
conversationHistory (CouncilMessage[]) → provider.chat(messages, options)
```

v0.5.1 change: extend `respond()` signature with an optional **`snapshotPrefix?: string`** parameter. When non-empty, construct a synthetic `CouncilMessage` at the head of the messages array:

```typescript
const prefixMsg: CouncilMessage = {
  id: `snapshot-prefix-${Date.now()}`,
  role: 'human',
  content: `Prior segment summary (from /councilreset):\n\n${snapshotPrefix}\n\n---\n\nThe conversation below is the new segment. Treat the summary above as shared context.`,
  timestamp: Date.now(),
};
const withPrefix = [prefixMsg, ...conversationHistory];
```

This message rides the **existing** `ProviderMessage[]` transformation in each provider (`role: 'human'` → `'user'` for all three providers). No provider-specific code paths.

**Trade-off, explicit:** v0.4's `systemPromptParts` ephemeral cache marker is **not applied** to the snapshot. Every resumed turn re-pays the snapshot tokens as input. This is deliberate for v0.5.1 — the feature-complete version trades cache efficiency for multi-provider correctness. v0.5.2 will add a Claude-only `systemPromptParts` fast path that preserves caching, gated on `provider.name === 'claude'`.

### 4.4 `Deliberation` refactor (`src/council/deliberation.ts`)

`SessionState.conversationHistory: CouncilMessage[]` → `SessionState.segments: HistorySegment[]`. Same migration as the pre-Codex plan, but scope confirmed:

```typescript
interface SessionState {
  segments: HistorySegment[];
  currentParticipants: string[];
  turnManager: TurnManager;
  antiSycophancy: AntiSycophancyEngine;
  pendingPatternInjection: { targetAgent: string; prompt: string } | null;
  critiqueLog: CritiqueSessionLog;
  blindReviewSessionId: string | null;
  currentTopic: string;
}

interface HistorySegment {
  startedAt: string;
  endedAt: string | null;
  messages: CouncilMessage[];
  snapshotId: string | null;
}
```

Public per-thread methods on `DeliberationHandler`: `getSegments(tid)`, `getCurrentSegmentMessages(tid)`, `sealCurrentSegment(tid, snapshotId)`, `openNewSegment(tid)`, `getSnapshotPrefix(tid)`, `getBlindReviewSessionId(tid)`, `getCurrentTopic(tid)`.

All three existing `session.conversationHistory.push(...)` sites (confirmed lines 139, 217, 403) push into `session.segments[last].messages`.

Every in-deliberation `worker.respond(...)` call site calls `this.getSnapshotPrefix(threadId)` and passes the result as the new `snapshotPrefix` arg.

**Reset-flow facilitator call does NOT pass the snapshot prefix** (see §4.1 step 2, Codex finding #2 mitigation).

### 4.5 `currentTopic` wiring

Codex correctly identified that `currentTopic` has no existing source. Fix: add `topic` to `EventMap['deliberation.started']`. `DeliberationHandler` already calls `runDeliberation(threadId, message, intent, complexity)` internally; inspect the first user message content or the classifier output (whichever is already available at the `deliberation.started` emit site, `deliberation.ts:273`) and include it in the emit payload. Subscribe in the same handler to assign `session.currentTopic`.

If no topic is resolvable at the emit site, use the first message's content truncated to 80 chars. This is a minimal surface change.

### 4.6 `blindReviewSessionId` wiring

Codex correctly identified the emit site. `blind-review.started` is emitted from `src/council/deliberation.ts:466`, not from `blind-review.ts`. Session id computation (`${threadId}:${session.startedAt}`) currently happens inside `BlindReviewStore`.

Fix: when `BlindReviewStore.create()` returns, its result object includes `session.sessionId` (add it to `CreateResult`). The existing `deliberation.ts` emit site (line 466) then includes `sessionId` in the `blind-review.started` payload. `DeliberationHandler` also subscribes to its own `blind-review.started` event (or mutates `session.blindReviewSessionId` in the same block where the emit happens — simpler) and to `blind-review.revealed` to clear it.

### 4.7 Commands

- `/councilreset` — wired in `src/telegram/bot.ts` (Telegram) and `src/adapters/cli-commands.ts` (CLI). **Codex finding #8:** the pre-Codex plan targeted wrong files (`src/adapters/telegram.ts`); correct targets confirmed.
  - On `BlindReviewActiveError`, reply with the error message only (no reset).
  - On success, reply with `Sealed segment N: M decision(s), K open question(s). Starting next segment.`
- `/councilhistory` — lists all snapshots for the active thread with sealed-at timestamp and metadata counts.

## 5. Data model

### 5.1 `session_reset_snapshots`

Stored in `data/council.db`, new table alongside existing `BlindReviewDB` tables.

| Column | Type | Notes |
|---|---|---|
| `snapshot_id` | TEXT PRIMARY KEY | UUIDv4 |
| `thread_id` | INTEGER NOT NULL | Telegram thread / CLI session binding |
| `segment_index` | INTEGER NOT NULL | 0-indexed, `(thread_id, segment_index)` unique |
| `sealed_at` | TEXT NOT NULL | ISO-8601 |
| `summary_markdown` | TEXT NOT NULL | Source of truth for prepend |
| `open_questions_count` | INTEGER NOT NULL DEFAULT 0 | Metadata index |
| `decisions_count` | INTEGER NOT NULL DEFAULT 0 | Metadata index |
| `blind_review_session_id` | TEXT | Always `NULL` in v0.5.1 (reset is guarded against active blind-review). Kept for v0.5.3 carry-forward. |

Indexes: `(thread_id, segment_index)` unique, `(thread_id, sealed_at DESC)` for `/councilhistory`.

### 5.2 In-memory, not persisted

`HistorySegment.messages` is **not** persisted across process restart in v0.5.1. Restart resets segment state; snapshot audit trail in DB survives. Durable segment storage = v0.5.2.

## 6. Summary markdown format + cache-hit trade-off

Facilitator reset prompt constrains output to exactly four `##` sections:

```markdown
## Decisions
- <decision 1>

## Open Questions
- <question 1>

## Evidence Pointers
- <citation or prior-turn reference>

## Blind-Review State
<free text: current agent tier assignments + PVG rotation state, or "none">
```

**Determinism:** facilitator called via `AgentWorker.respondDeterministic(messages, 'synthesizer', { temperature: 0 })`. Prompt template is identical across invocations (same topic-independent header bytes). Output variance is model-noise only, tolerable because the snapshot is **read** by downstream agents as plain message content — not hashed, not cached.

**Cache hit not claimed in v0.5.1.** The prepend path in §4.3 surfaces the snapshot as a regular user-role message. Anthropic ephemeral cache markers are on `systemPromptParts`, not on `messages[*].content`. v0.5.2 adds a Claude-only optimisation that emits the snapshot as a cached system part in addition to (or replacing) the prepend. v0.5.1's correctness claim is **multi-provider carry-forward**, not cache efficiency.

## 7. Exit criteria (v0.5.1, revised)

- [ ] `/councilreset` ships on CLI and Telegram.
- [ ] Snapshot persists via `ResetSnapshotDB`. Markdown + metadata readable via `/councilhistory`.
- [ ] Next-turn invocation after reset delivers snapshot as the first user-role message in `conversationHistory` for **every provider** (claude, openai, gemini). Verified in integration test by asserting the provider's `chat()` receives the snapshot text at `messages[0].content`.
- [ ] Reset fails fast and cleanly when blind-review is active. Verified in integration test.
- [ ] Seal-after-snapshot failure rolls back the snapshot DB row. Verified in unit test.
- [ ] `docs/LONG_RUNNING_COUNCIL.md` explains the reset flow, the multi-provider prepend choice, and notes cache-hit is deferred.
- [ ] All existing tests still pass. New tests follow TDD.

## 8. Open risks (v0.5.1)

1. **Summary bytes drift across runs** (facilitator output varies even at `temperature: 0` due to model non-determinism and topic-content interleaving). Acceptable — the snapshot is text content, not a cache key. Users rerunning the same topic from scratch will get slightly different summaries; this is expected.
2. **Prepend doubles prompt length on the first post-reset turn** (roughly +summary size in tokens for every agent response). Acceptable — the point of reset is to stop sending N prior turns. Prepend is O(summary) ≪ O(prior turns) for sessions that triggered reset.
3. **`BlindReviewActiveError` user-facing message must not leak session id to Telegram group chat.** Message template uses `/blindreview reveal` and `/cancelreview` by name only.
4. **Concurrent `/councilreset` on the same thread** (e.g., user double-taps the command). Mitigation: `DeliberationHandler` adds a per-thread reset-in-flight flag. Second concurrent reset returns "reset in progress" error without triggering a second facilitator call. Flag cleared on success or failure.
