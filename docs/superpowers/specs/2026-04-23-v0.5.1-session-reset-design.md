# v0.5.1 Session Reset — Design

**Version target**: v0.5.1
**Theme**: Long-running council sessions (Anthropic *Harness Design for Long-Running Applications*, 2026-04)
**Roadmap anchor**: `agent-council-ROADMAP.md` §0.5.1
**Date**: 2026-04-23

---

## 1. Problem

Council sessions in a single Telegram chat (or CLI session) accumulate an ever-growing `CouncilMessage[]` history. v0.4.0 prompt caching mitigates token cost but not semantic drift — past ~100 turns, agent responses start echoing earlier rounds and sessions "lose the plot". There is no user-facing primitive to mark a conceptual boundary ("we're done with topic A, starting topic B") while preserving audit.

## 2. Goal

Ship a user-triggered `/councilreset` command that:

1. Produces a structured session summary (decisions, open questions, evidence pointers, blind-review state) and persists it.
2. On next turn, feeds the summary as a stable cached prefix instead of resending prior turns.
3. Keeps prior turns readable via `/councilhistory` for audit — but not resent to agents.
4. Surfaces a passive suggestion when the largest single-invocation `input_tokens` passes 80% of the model's `max_input_tokens`.

**Non-goals** (roadmap explicit):

- Automatic compaction / auto-reset. Reset is always user-triggered.
- Per-agent snapshot variants (all agents share one snapshot per reset).
- Cross-chat / cross-session snapshot reuse (future work, v0.5.3).

## 4. Components

### 4.1 `SessionReset` (`src/council/session-reset.ts`)

Orchestrates the reset flow:

1. Calls facilitator with a reset-summary prompt (the existing facilitator agent, dedicated prompt, not a new agent) to produce structured markdown.
2. Parses minimal metadata from the markdown (counts of `## Decisions` / `## Open Questions` entries, current blind-review session_id from deliberation state).
3. Persists via `ResetSnapshotDB.recordSnapshot()` in a transaction.
4. Calls `Deliberation.sealCurrentSegment(snapshotId)` and `Deliberation.openNewSegment()`.

Returns `{ snapshotId, summaryMarkdown, metadata }` for display to the user.

**Failure modes**:

- Facilitator call fails → reset aborts, history unchanged, error surfaced to user ("reset failed, session continues").
- DB write fails → wrapped in single transaction with segment seal; both roll back.

### 4.2 `ResetSnapshotDB` (`src/storage/reset-snapshot-db.ts`)

Pattern-matches `BlindReviewDB` (constructor, `migrate()` with `CREATE TABLE IF NOT EXISTS`, `:memory:` branch for tests, transactional writes). Schema in §5.

### 4.3 `ModelRegistry` (`src/worker/model-registry.ts`)

```ts
interface ModelInfo {
  id: string;
  maxInputTokens: number;
}

class ModelRegistry {
  private cache = new Map<string, ModelInfo>();
  constructor(private client: Anthropic) {}
  async get(modelId: string): Promise<ModelInfo>;  // lazy retrieve + cache
}
```

- First `get(id)` calls `client.models.retrieve(id)` → stores `{ id, maxInputTokens: response.max_input_tokens }`.
- Subsequent calls hit in-memory cache (process lifetime).
- **No fallback on failure** — retrieve error propagates. Rationale: if the Models API is unreachable, the council itself cannot run, so context-threshold inaccuracy is a secondary concern.
- Test injection: provider layer takes a `ModelRegistry`-shaped interface, tests supply a stub.

### 4.4 `ContextUsageTracker` (`src/council/context-usage.ts`)

- Stateful per-deliberation.
- After each `ClaudeProvider.chat()` response, receives `{ model, inputTokens }`.
- Updates `peakInputTokens` per model.
- On update: compares `peakInputTokens / registry.get(model).maxInputTokens`; if ≥ 0.80 and not already emitted **for this segment**, emits an event (CLI/Telegram adapter prints a one-line hint). Threshold re-arms after `openNewSegment()`.

### 4.5 Provider changes (`src/worker/providers/claude.ts` + `base.ts`)

Extend `ProviderResponse.tokensUsed` from `{ input, output }` to:

```ts
tokensUsed: {
  input: number;
  output: number;
  cacheCreationInput?: number;
  cacheReadInput?: number;
}
```

Populated from `response.usage.cache_creation_input_tokens` / `cache_read_input_tokens`. Required for v0.5.1 exit criteria (verify summary is actually caching).

### 4.6 `Deliberation` refactor (`src/council/deliberation.ts`)

`this.history: CouncilMessage[]` → `this.segments: HistorySegment[]`.

```ts
interface HistorySegment {
  startedAt: string;
  endedAt: string | null;
  messages: CouncilMessage[];
  snapshotId: string | null;  // filled when segment is sealed
}

private get currentSegment() { return this.segments.at(-1)!; }

sealCurrentSegment(snapshotId: string): void;
openNewSegment(): void;
```

All three existing `history.push(...)` call sites (human message, agent response, facilitator intervention — lines 216, 296, 402) push into `currentSegment.messages` instead.

Agent invocations build `messages` from `currentSegment.messages` only. When `currentSegment` has an immediate predecessor with `snapshotId !== null`, the predecessor's snapshot markdown is loaded (via `ResetSnapshotDB.get`) and injected as a cached `SystemPromptPart` (`{ text: summaryMd, cache: true }`) at the top of the system prompt.

### 4.7 Commands

- `/councilreset` — CLI (`src/adapters/cli-commands.ts`) + Telegram (new handler, pattern-match existing `/pvgrotate` handler). Calls `SessionReset.reset(deliberation)`. On success, replies with segment number + metadata summary ("Segment 0 sealed: 3 decisions, 2 open questions. Starting segment 1.").
- `/councilhistory` — lists all segments with timestamps, decision counts, and "reveal" action to dump full segment transcript. CLI prints inline; Telegram sends as paginated message.

## 5. Data model

### 5.1 `session_reset_snapshots`

Stored in `data/council.db` (same DB as `BlindReviewDB`, new table).

| Column | Type | Notes |
|---|---|---|
| `snapshot_id` | TEXT PRIMARY KEY | UUIDv4 |
| `thread_id` | INTEGER NOT NULL | Telegram thread / CLI session binding |
| `segment_index` | INTEGER NOT NULL | 0-indexed, `(thread_id, segment_index)` unique |
| `sealed_at` | TEXT NOT NULL | ISO-8601 |
| `summary_markdown` | TEXT NOT NULL | Source of truth for cached prefix |
| `open_questions_count` | INTEGER NOT NULL DEFAULT 0 | Metadata index |
| `decisions_count` | INTEGER NOT NULL DEFAULT 0 | Metadata index |
| `blind_review_session_id` | TEXT | FK-ish to `blind_review_sessions.session_id`, nullable |

Indexes: `(thread_id, segment_index)` unique, `(thread_id, sealed_at DESC)` for `/councilhistory`.

Not normalised: counts may drift from markdown content. Markdown is the authority; counts are list-view hints.

### 5.2 `HistorySegment` (in-memory only for v0.5.1)

Not persisted across process restarts in v0.5.1. Process restart mid-session resets segment state. This is acceptable because the snapshot table preserves the audit record; only the live `currentSegment.messages` is volatile. Durable segment storage is v0.5.2+ work.

## 6. Summary markdown format

Facilitator reset prompt constrains output to exactly four `##` sections:

```markdown
## Decisions
- <decision 1>
- <decision 2>

## Open Questions
- <question 1>

## Evidence Pointers
- <citation or prior-turn reference>

## Blind-Review State
<free text: current agent tier assignments + PVG rotation state, or "none">
```

Metadata extraction is a single-pass line parser: count `-` bullet lines under `## Decisions` and `## Open Questions`; read blind-review `session_id` from `Deliberation.blindReviewSessionId` state (not from markdown).

Stable prefix property: facilitator temperature is clamped to 0 for reset-summary prompts and the section headers are fixed. Prefix bytes identical across otherwise-identical invocations → cache hits.

## 7. Context threshold detection

- On every provider response (all agents, facilitator), `ContextUsageTracker.record(model, inputTokens)` is called.
- Tracker maintains `peakPerModel: Map<string, number>`.
- After update, compute `ratio = peakPerModel[model] / registry.get(model).maxInputTokens`.
- If `ratio >= 0.80` and `!hintEmittedThisSegment`:
  - Emit event `context-usage-high` with `{ model, peakInputTokens, maxInputTokens, ratio }`.
  - Adapter (CLI / Telegram) prints one line: `⚠ Context usage 82% (claude-opus-4-7 peak 164000/200000 tokens). Consider /councilreset.`
  - `hintEmittedThisSegment = true`.
- `openNewSegment()` clears `peakPerModel` and `hintEmittedThisSegment`.

Test fixtures override `ModelRegistry` with synthetic `maxInputTokens` (e.g. 10_000) to drive threshold deterministically.

## 8. Exit criteria (v0.5.1)

- [ ] `/councilreset` ships on CLI and Telegram.
- [ ] Snapshot persists via `ResetSnapshotDB`; markdown + metadata readable via `/councilhistory`.
- [ ] Next-turn invocation after reset sends summary as cached system-prompt part. Verified in an integration test by asserting `response.usage.cache_creation_input_tokens > 0` on first use and `cache_read_input_tokens > 0` on second use.
- [ ] `ContextUsageTracker` emits hint ≥80% with correct ratio; test drives it via `:memory:` DB + mocked `ModelRegistry`.
- [ ] `ProviderResponse.tokensUsed` returns `cacheCreationInput` / `cacheReadInput` for Claude provider.
- [ ] `docs/LONG_RUNNING_COUNCIL.md` explains the reset flow with one empirical token-count example.
- [ ] All existing tests still pass. New tests follow TDD (each failing first).

## 10. Open risks

1. **Facilitator markdown drift**: if facilitator disobeys the four-section template, metadata parsing returns zero counts. Mitigation: in v0.5.1, metadata drift is cosmetic (counts wrong, markdown still correct). Post-v0.5.1: add a retry-with-correction loop.
2. **`max_input_tokens` missing from older model responses**: SDK docs confirm the field; if a specific model returns `null`/`undefined`, `ModelRegistry` throws. Acceptable — fail fast on unsupported model.
3. **Prompt cache invalidation**: any whitespace change in the summary or upstream system prompt parts breaks cache. Mitigation: snapshot markdown stored exactly as written, no post-processing; upstream system prompt parts already have cache markers from v0.4.
4. **Summary prompt temperature**: clamped to 0 to keep prefix bytes stable. Trades creativity for cacheability — correct for this module.

## 11. Decision log

| # | Decision | Why |
|---|---|---|
| 1 | Q1 → C (markdown blob + 3 metadata counts + blind-review FK) | Prefix cacheability requires stable blob; metadata enables `/councilhistory` list view without parsing |
| 2 | Q2 → A (sealed-archive segments) | Clean before/after boundary; directly enables v0.5.3 carry-forward-blind-review-only via segment + partial snapshot |
| 3 | Q3a → B1 (Models API, no fallback) | `max_input_tokens` field confirmed in Models API schema; single source of truth, fail-fast on API unavailability |
| 4 | Q3b → B (peak single-invocation input_tokens) | Context window is a per-call limit; "cumulative" in roadmap was directional, peak is the correct metric |
| 5 | Q3c → A (passive hint only) | Roadmap explicitly rejects auto-compaction; facilitator-injected suggestion would pollute discussion |
| 6 | Name `session-reset` / `ResetSnapshotDB` / `session_reset_snapshots` | Avoids collision with existing `src/memory/session-summary.ts` (agent-level per-session memory artifact, different semantics) |
| 7 | `HistorySegment[]` not persisted in v0.5.1 | Snapshots preserve audit; live segment volatility acceptable; durable storage scoped to v0.5.2 |

## 12. Roadmap wording amendment

Roadmap §0.5.1 currently says "auto-suggest reset when cumulative turn tokens pass a threshold". Update to "auto-suggest reset when peak single-invocation `input_tokens` passes 80% of the model's `max_input_tokens`." Rationale: cumulative double-counts (each agent sees prior agents' responses), and context window is a per-invocation constraint.
