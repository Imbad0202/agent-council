## 0. TL;DR

Stop the **provider-hang retry storm** by threading `AbortSignal` from
per-attempt timeouts down to the four LLM SDKs. Caller-initiated thread
cancellation (`/councilreset`/`/councildone` aborting in-flight peer chats)
is **out of scope** for v0.5.3 — the existing refuse-while-in-flight design
from v0.5.1 round-7..12 codex review stays untouched.

Approach: add optional `signal: AbortSignal` field to `ChatOptions`
(decision 1B). Wire 4 provider implementations to forward `signal` to their
SDKs. Wire two callsite layers — `invokeProviderForArtifact` (synthesis hot
path) and the facilitator-intervention `raceWithTimeout` site — to add
native abort alongside the existing `Promise.race` backstop. Add optional
`signal` as 7th positional param on `AgentWorker.respond()` for the
facilitator-intervention path **only** (peer-turn callers continue to
pass nothing). Don't touch IntentGate, dispatcher, memory layer, or
`respondDeterministic()`.

## 0.3 Timeout backstop must be preserved (codex round-3 P1 → user 2026-04-29 12:28 → option A)

Round-3 codex flagged that round-2's `invokeProviderForArtifact` body
removed `Promise.race` and relied on `provider.chat` honoring
`options.signal`. **`signal` is best-effort; the timeout race is a
deterministic safety net.** A noncompliant provider (or a third-party
`LLMProvider` impl that ignores `signal`) would leave `/councildone` and
facilitator intervention with worse-than-pre-v0.5.3 behavior: no caller
exit at all.

**Round-3 invariant (v0.5.3 scope)**: the **two existing in-scope timeout
sites** keep their caller-side timeout race as the deterministic exit.
The signal is added as an *additional* mechanism that also tells the SDK
to abort, freeing server-side resources where the provider supports it.
Race and signal run in parallel, not in substitution.

The invariant is scoped to v0.5.3's two sites:
- `invokeProviderForArtifact` (§3.3)
- Facilitator-intervention callsite (§5.1 site 5)

Round-4 P3-2 note: this is **not** a global guarantee for future timeout
sites added after v0.5.3. v0.5.4+ should either keep this invariant by
convention or extract a shared helper (e.g.
`raceWithTimeoutAndSignal(p, ms, ctrl)`) that bakes the pattern in. Out
of scope here.

`§9.2` provider-implementor migration is updated accordingly: ignoring
`signal` falls back to pre-v0.5.3 caller-side timeout behavior, which is
the existing contract — true now.

## 0.2 Google abort is client-only (codex round-2 P1 → user 2026-04-29 12:17 → option C)

`@google/genai` documents `abortSignal` as **client-side only**: the HTTP
request is cancelled but server-side generation continues and continues to
bill (`node_modules/@google/genai/dist/genai.d.ts:4339` per round-2 review).

This breaks v0.5.3's headline §2 invariant "1× per-attempt budget × MAX_ATTEMPTS
sequential, not 4× concurrent" for Google users. Three options surfaced:

- A) Google timeout = hard fail, no retry (simplest; sacrifices retry for transient Google issues)
- B) Spec admits the asymmetry (honest; documentation gets messy)
- C) **Hybrid (chosen)**: on the **first** timeout from Google specifically, throw
  `GoogleProviderTimeoutError` (subclass of `ProviderTimeoutError`, see
  §6.1.1) directly, without further retry. Other providers retry up to
  `MAX_ATTEMPTS = 4` as normal. Rationale:
  1. Honest: Google users see one clear error, not silently burned tokens
  2. Bounded: server-side generation continues but at most once per
     `/councildone` invocation, not 4×
  3. Symmetric to current `isHardFail` design: just adds a provider-specific
     hard-fail predicate
  4. No retry-loop logic change beyond classification

The §0.1 scope decision (refuse design preserved) stands.

## 1. Problem (narrowed)

### 1.1 The retry storm (v0.5.2 spec §5)

`invokeProviderForArtifact` uses `Promise.race(provider.chat, timeoutPromise)`.
When the timeout wins, `provider.chat` keeps running in the background. Under
`MAX_ATTEMPTS = 4` with all attempts timing out at 30s:

- Up to 4 concurrent in-flight requests against the same provider
- Up to 4× per-attempt token cost
- Rate-limit headroom consumed by orphaned attempts
- Caller-visible bound stays at ~127s, but **token cost** grows ~4×

### 1.2 The facilitator intervention twin

`DeliberationHandler` wraps `evaluateIntervention()` with
`raceWithTimeout(p, DEFAULT_FACILITATOR_INTERVENTION_TIMEOUT_MS=30000, ...)`
(`src/council/deliberation.ts:47-50`). Same pattern, same leak: timeout
unblocks the deliberation loop but the underlying `worker.respond()` keeps
running. v0.5.2 P1-B fixed the *mutation* race; the *resource* leak survived.

### 1.3 What this spec does NOT solve

- User-initiated `/councilreset` / `/councildone` aborting peer chats — v0.5.4
- Memory consolidation cancellation — v0.5.4
- Dispatcher (`Executor.execute`) cancellation — v0.5.4
- IntentGate `llmClassify` cancellation — v0.5.4 (current swallow-and-default
  behavior is intentional)
- Streaming responses — not used in agent-council yet
- `provider.summarize()` cancellation — v0.5.4 (interface change)

## 2. Goal & non-goals

### Goal

For the **two specific paths** in scope (artifact synthesis retry, facilitator
intervention), the per-attempt caller-side timeout becomes a real network
abort, such that:

1. When the per-attempt timer fires, the underlying HTTP request is aborted
   at the SDK / fetch level.
2. No new client attempts start while a prior attempt's request is still
   in-flight client-side.
3. Token cost upper bound (Claude / OpenAI / Custom):
   `1× per-attempt budget × MAX_ATTEMPTS` sequential, not 4× concurrent.
4. Token cost upper bound (Google, per §0.2 option C): **`1× per-attempt
   budget`** — first timeout fails the synthesis, no further attempts.
   Server-side generation may still complete and bill once.
5. Facilitator intervention timeout no longer leaves orphan provider chats
   client-side. Same Google asymmetry applies.

### Non-goals (ALL deferred to v0.5.4+)

- User-initiated cancellation
- Thread-scoped `AbortController` map on `DeliberationHandler`
- Peer-turn callers of `AgentWorker.respond()` passing a signal (round-4
  P2-4: spec **does** add optional 7th positional param; only
  facilitator-intervention path uses it; peer-turn callers pass nothing)
- `respondDeterministic()` signature change
- IntentGate cancellation
- Memory layer cancellation
- Dispatcher cancellation
- `provider.summarize()` signature change
- Cancellation telemetry / metrics
- `/councilcancel` user command

### 3.2 Two abort sources, single merge primitive

Each callsite that wants both a per-attempt timeout AND honors caller's
`options.signal` merges them via `AbortSignal.any`:

```ts
// at callsite
const timeoutCtrl = new AbortController();
const timeoutId = setTimeout(() => timeoutCtrl.abort(new TimeoutReason()), perAttemptMs);
try {
  const merged = options.signal
    ? AbortSignal.any([options.signal, timeoutCtrl.signal])
    : timeoutCtrl.signal;
  const response = await provider.chat(messages, { ...options, signal: merged });
  return response;
} finally {
  clearTimeout(timeoutId);
}
```

`AbortController` + manual setTimeout (rather than `AbortSignal.timeout`) is
chosen so we control the **reason** carried on the signal. This is the fix
for round-1 P2 timeout-cause-detection: when `merged.reason instanceof
TimeoutReason`, surface as `ProviderTimeoutError`; otherwise the caller's
abort takes precedence.

```ts
// new in src/abort-utils.ts (round-2 P3: top-level, not under council/, so
// src/worker/providers/custom.ts can import without inverting layers)
export class TimeoutReason extends Error {
  constructor(public readonly perAttemptMs: number) {
    super(`Per-attempt timeout after ${perAttemptMs}ms`);
    this.name = 'TimeoutReason';
  }
}

export function isAbortError(err: unknown): boolean {
  if (!err || typeof err !== 'object') return false;
  const e = err as { name?: string; code?: string; constructor?: { name?: string } };
  // DOM-style and Node-style
  if (e.name === 'AbortError') return true;
  if (e.code === 'ABORT_ERR') return true;
  // Pinned SDK-specific abort classes (round-1 P1-2 fix)
  // Anthropic SDK: APIUserAbortError extends APIError; runtime name is "Error"
  // OpenAI SDK: APIUserAbortError, same shape
  if (e.constructor?.name === 'APIUserAbortError') return true;
  return false;
}

export function mergeSignals(
  ...signals: (AbortSignal | undefined)[]
): AbortSignal | undefined {
  const present = signals.filter((s): s is AbortSignal => s !== undefined);
  if (present.length === 0) return undefined;
  if (present.length === 1) return present[0];
  return AbortSignal.any(present);
}

export function isTimeoutAbort(signal: AbortSignal): boolean {
  return signal.aborted && signal.reason instanceof TimeoutReason;
}
```

### 3.3 Add native abort while keeping `Promise.race` backstop (artifact path)

Per §0.3, the timeout race is preserved as a deterministic backstop. The
new `signal` runs in parallel, telling the SDK to also abort.

```ts
// v0.5.3 invokeProviderForArtifact
export async function invokeProviderForArtifact(
  provider: LLMProvider,
  messages: ProviderMessage[],
  options: ChatOptions,
  perAttemptTimeoutMs: number = PER_ATTEMPT_TIMEOUT_MS,
): Promise<ProviderResponse> {
  const timeoutCtrl = new AbortController();
  const merged = mergeSignals(options.signal, timeoutCtrl.signal);

  let timeoutId: ReturnType<typeof setTimeout> | undefined;
  try {
    return await Promise.race([
      provider.chat(messages, { ...options, signal: merged }),
      new Promise<never>((_, reject) => {
        timeoutId = setTimeout(() => {
          // Abort the SDK request (best-effort) AND reject the race deterministically.
          timeoutCtrl.abort(new TimeoutReason(perAttemptTimeoutMs));
          reject(new ProviderTimeoutError(perAttemptTimeoutMs));
        }, perAttemptTimeoutMs);
      }),
    ]);
  } catch (err) {
    // Round-2 P2 fix: Node fetch rejects with signal.reason when reason is
    // an Error. So `err` may BE the TimeoutReason instance for CustomProvider.
    // (But the race already throws ProviderTimeoutError before this; this
    // catch only sees TimeoutReason if the SDK rethrew it before the
    // setTimeout fired — order-of-microtasks edge case.)
    if (err instanceof TimeoutReason) {
      throw new ProviderTimeoutError(perAttemptTimeoutMs);
    }
    if (err instanceof ProviderTimeoutError) {
      throw err; // race won; pass through
    }
    if (isAbortError(err)) {
      if (merged?.aborted && merged.reason instanceof TimeoutReason) {
        throw new ProviderTimeoutError(perAttemptTimeoutMs);
      }
      // External cancellation (caller signal) — surface as-is.
      throw err;
    }
    throw err;
  } finally {
    if (timeoutId !== undefined) clearTimeout(timeoutId);
  }
}
```

**Why both race AND signal**:

| Provider behavior | What happens at timeout |
|---|---|
| Honors `signal` (Claude / OpenAI / Custom) | race rejects `ProviderTimeoutError`; SDK aborts request; both client and server stop (server may continue for Google) |
| Ignores `signal` (noncompliant or third-party impl) | race rejects `ProviderTimeoutError`; underlying request orphaned just like pre-v0.5.3 — no regression |
| Honors `signal` AND fetches reject before setTimeout fires | catch sees `TimeoutReason` or `AbortError`; reclassified to `ProviderTimeoutError` per cause table |

**Cause classification table** (rounds 2-3 fix):

| Catch error shape | merged.reason | Throws |
|---|---|---|
| `ProviderTimeoutError` (race won) | n/a | re-throw |
| `TimeoutReason` direct (rare; SDK rethrew) | `TimeoutReason` | `ProviderTimeoutError` |
| SDK `APIUserAbortError` / DOM AbortError | `TimeoutReason` | `ProviderTimeoutError` |
| SDK `APIUserAbortError` / DOM AbortError | non-Timeout (caller's reason) | re-throw original (caller cancellation) |
| Non-abort error | n/a | re-throw |
| `TimeoutError` DOMException (CustomProvider's internal 60s timeout) | n/a | round-4 P3-1: cannot win under 30s artifact budget; if it ever did fire, falls into "non-abort error" row above, re-thrown as-is. v0.5.4 may add explicit handling if budget becomes configurable. |

```ts
// ONE timer drives both (a) the abort signal AND (b) the race-reject.
// Eliminates round-4 P2-1 (abort-vs-reject race) and P2-2 (zombie timer).
const interventionCtrl = new AbortController();
let timeoutId: ReturnType<typeof setTimeout> | undefined;
let timedOut = false;

let result: FacilitatorInterventionResult | null = null;
try {
  result = await Promise.race([
    this.facilitatorIntervention.evaluateIntervention(threadId, interventionCtrl.signal),
    new Promise<null>((_, reject) => {
      timeoutId = setTimeout(() => {
        timedOut = true;
        // Reject the race FIRST so timeout is the deterministic winner
        // even if the hook is about to settle synchronously after this.
        reject(new TimeoutReason(DEFAULT_FACILITATOR_INTERVENTION_TIMEOUT_MS));
        // Then abort the signal (best-effort, releases SDK request).
        interventionCtrl.abort(new TimeoutReason(DEFAULT_FACILITATOR_INTERVENTION_TIMEOUT_MS));
      }, DEFAULT_FACILITATOR_INTERVENTION_TIMEOUT_MS);
    }),
  ]);
  // Round-4 P2-1 belt-and-suspenders: if hook somehow settled simultaneously
  // with timeout firing, force null when timedOut==true.
  if (timedOut) result = null;
} catch (err) {
  // ALL errors swallowed/logged here, matches v0.5.2 current behavior of
  // `raceWithTimeout(...).catch(() => null)`. signal is best-effort;
  // race ensures deterministic exit.
  console.error('[facilitator] intervention failed:', err);
  result = null;
} finally {
  if (timeoutId !== undefined) clearTimeout(timeoutId);
}

// process result (null = skip intervention, existing behavior)
```

**Single-timer design**: one setTimeout handles both the race-reject
(authoritative exit on timeout) and the abort-dispatch (best-effort SDK
cleanup). `reject(...)` runs synchronously before `abort(...)` so once
the timeout callback **begins executing**, timeout is authoritative —
hook completions processed before that callback fires are acceptable
pre-timeout settles. Plus `timedOut` guard re-asserts on the resolve
path. `finally { clearTimeout }` matches existing `raceWithTimeout` hygiene.

(Round-5 P3 note: this is not "same-tick" determinism in the absolute
sense; it's "once the timeout callback runs, timeout wins". The race
honors whichever side's promise settles first per the JS event loop's
normal scheduling.)

`raceWithTimeout` helper itself stays in the codebase if used elsewhere
(verify by grep before removing).

**Why this is still additive**:
- All new params optional, default undefined
- Existing peer-turn `runDeliberation` callsites of `respond()` pass nothing
- Existing `FacilitatorAgent.evaluateIntervention(threadId)` pre-v0.5.3
  callers compile (signal defaults to undefined)

`respondDeterministic()` is **not** changed in v0.5.3 (used only by
`SessionReset` for facilitator summary, which is v0.5.4 scope).

## 6. Retry-storm fix (the headline)

### 6.1 `isHardFail` picks up provider-aware timeout policy

```ts
// src/council/artifact-invoke.ts — modified
export function isHardFail(err: unknown, provider?: LLMProvider): boolean {
  // Round-2 §0.2 option C: Google's abortSignal is client-side only;
  // server-side generation continues to bill. Treat first timeout as
  // hard fail to bound cost at 1× per-attempt budget instead of N×.
  if (err instanceof ProviderTimeoutError && provider?.name === 'google') {
    return true;
  }
  if (err instanceof ProviderTimeoutError) return false;
  if (err instanceof EmptyResponseError) return false;
  if (err instanceof Error && 'status' in err) {
    const status = (err as { status: number }).status;
    if (status === 429) return false;
    if (status >= 500 && status < 600) return false;
    if (status >= 400 && status < 500) return true;
  }
  return false;
}
```

Note: `LLMProvider.name` field already exists on every provider impl
(`base.ts:4` `abstract readonly name: string`); 'claude' / 'openai' /
'google' / 'custom'.

### 6.1.1 Google error contract (round-3 P2-3 + round-4 P2-3 fix)

Round-4 P2-3 caught that `ArtifactService` has no `userMessage` formatter
and no `this.deps.provider` (provider built locally via
`createProvider(synthesizerConfig)` at synthesis time). Adapters print
`err.message` directly. Solution: **bake the user-friendly message into a
specialized error class**, so `err.message` already carries the Google
disclosure. No formatter, no adapter changes, no `deps.provider`.

```ts
// src/council/artifact-errors.ts — new class
export class GoogleProviderTimeoutError extends ProviderTimeoutError {
  constructor(timeoutMs: number) {
    super(timeoutMs);
    this.name = 'GoogleProviderTimeoutError';
    this.message =
      `Google synthesis timed out after ${timeoutMs}ms. ` +
      `Google's API does not support cancelling server-side generation; ` +
      `to limit cost, retries are disabled for Google. ` +
      `Note: a single timed-out generation may still bill server-side. ` +
      `Try /councildone again later, or switch synthesizer to Claude/OpenAI.`;
  }
}
```

Throw site is `invokeWithRetry`, after `isHardFail(err, provider)` returns
true:

```ts
// src/council/artifact-invoke.ts — invokeWithRetry catch block
if (isHardFail(err, provider)) {
  // Round-4 P2-3: provider-specific error class for cleaner UX.
  if (err instanceof ProviderTimeoutError && provider.name === 'google') {
    throw new GoogleProviderTimeoutError(err.timeoutMs);
  }
  throw err;
}
```

Adapters (CLI `cli-commands.ts:240`, Telegram `bot.ts:236`) require **zero
changes** — both already do `err instanceof Error ? err.message : String(err)`.

**Why GoogleProviderTimeoutError extends ProviderTimeoutError**: existing
`isHardFail` checks `err instanceof ProviderTimeoutError`. Subclassing
keeps the predicate logic working without provider-name probing on every
hot path.

Wrapping in `SynthesisRetryExhaustedError` was rejected (round-3) because
that error implies "all 4 attempts ran", misleading for Google's 1-attempt

### 6.2 `invokeWithRetry` body

```ts
export async function invokeWithRetry(
  provider: LLMProvider,
  messages: ProviderMessage[],
  options: ChatOptions, // .signal currently always undefined; reserved for v0.5.4
): Promise<ProviderResponse> {
  let lastErr: unknown;
  for (let attempt = 1; attempt <= MAX_ATTEMPTS; attempt++) {
    if (attempt > 1) await sleep(SLEEPS_MS[attempt - 2]);
    try {
      const response = await invokeProviderForArtifact(
        provider, messages, options, PER_ATTEMPT_TIMEOUT_MS,
      );
      if (isProviderEmptyResponse(response.content)) {
        lastErr = new EmptyResponseError();
        continue;
      }
      return response;
    } catch (err) {
      lastErr = err;
      // Provider-aware: Google ProviderTimeoutError is hard-fail per §0.2.
      if (isHardFail(err, provider)) {
        // Round-5 P2 fix: align §6.2 snippet with §6.1.1 wrapping.
        if (err instanceof ProviderTimeoutError && provider.name === 'google') {
          throw new GoogleProviderTimeoutError(err.timeoutMs);
        }
        throw err;
      }
      // Note: external-cancellation early exit is v0.5.4 scope. v0.5.3
      // assumes no external signal is plumbed into options here.
    }
  }
  throw new SynthesisRetryExhaustedError(lastErr);
}
```

### 6.3 Per-attempt isolation property

Per-attempt `AbortController` lifecycle is owned **inside**
`invokeProviderForArtifact` (§3.3). Each call creates a fresh controller;
on return or throw, the `finally { clearTimeout }` releases the timer
reference and the controller goes out of scope. By the time attempt N+1
enters, attempt N's controller is unreferenced and its in-flight HTTP
request is either:
- (Claude / OpenAI / Custom) actually aborted at the network layer
- (Google) client-side aborted; server-side may still bill (§0.2 option C
  bounds this to one occurrence per `/councildone`)

**Properties**:
- Claude / OpenAI / Custom: at any moment, ≤1 in-flight HTTP request per
  `invokeWithRetry` call. Token cost upper bound: `1× per-attempt budget ×
  MAX_ATTEMPTS` sequential.
- Google: at any moment, ≤1 in-flight HTTP request **client-side** per
  `invokeWithRetry` call. Token cost upper bound: `1× per-attempt budget`
  client-side; server-side may bill once for the orphaned generation.
