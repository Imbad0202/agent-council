## 2. Invariants

### 2.1 `resetInFlight` always cleared

Existing `try { ... } finally { handler.setResetInFlight(threadId, false); }` in `session-reset.ts:117/242` is the single unwind point. Cancel paths throw `ResetCancelledError` from inside `try`, hit the same `finally`. **No new flag-clearing site needed.** Spec adds NO additional `setResetInFlight(false)` calls; auditing this prevents double-clear / use-after-clear regressions.

### 2.2 `safeDeleteSnapshot` covers cancel-after-DB-write

Cancel arriving between `db.recordSnapshot` (line 208) and the seal/open block goes through one of three race paths (§6 R1–R4). The two paths that need DB rollback (R2, R3) call the existing `safeDeleteSnapshot`. Cancel does NOT introduce a fourth code path; it shares the seal-failure / open-failure rollback machinery.

### 2.3 Second cancel is no-op, not error

If user fires `/councilcancel` twice (or fires it while reset is already settling), the second `controller.abort()` on an already-settled `AbortController` is a JS-spec no-op. `currentResetController` clear-on-finally (§3.3) means the second `/councilcancel` finds `null` and returns the friendly "no reset in progress" message (R6).

### 2.4 Thread isolation

`currentResetController` lives on `SessionState[threadId]`, NOT on `SessionReset` instance. A `/councilcancel` issued in thread A reads thread A's controller only — thread B reset is unaffected. Telegram chat_id and CLI thread-id naming match existing `setResetInFlight(threadId, ...)` invariant.

### 2.5 Validation error vs. cancel race (R5 detail)

If `respondDeterministic` returns successfully but `validateResetSummaryMarkdown` (line 178) throws `MalformedResetSummaryError`, AND a cancel arrives in the microtask gap between resolution and validation, **the validation error wins**. Reason: validation throws synchronously after the `await` resolves; cancel-via-AbortController only takes effect at the NEXT `await` point. There is no next `await` in the malformed-summary path before the throw. This is consistent with R4 ("cancel after open succeeds is too-late, no rollback").

Implementation on `DeliberationHandler` private `SessionState`:

```ts
interface SessionState {
  // ...existing fields
  currentResetController?: AbortController;

```

Lifecycle:
- `SessionReset.reset()` creates `new AbortController()` at start of `try` block (after all guards), calls `setCurrentResetController(threadId, controller)`
- In `finally`, after `setResetInFlight(false)`, calls `setCurrentResetController(threadId, null)`

### 4.1 detail — the race+abort pattern

```ts
// inside reset(), after all guards, after setResetInFlight(true):
const operationCtrl = new AbortController();
handler.setCurrentResetController(threadId, operationCtrl);

let timer: NodeJS.Timeout | undefined;
const merged = mergeSignals(options?.signal, operationCtrl.signal)!;
// merged is non-null because operationCtrl.signal is always present

try {
  const summaryPromise = (async () => {
    // ...existing reset body, lines 118–240, with one change:
    //    line 161: respondDeterministic(facilitatorMessages, 'synthesizer', merged)
    // Everything else identical.
  })();

  const racePromise = new Promise<never>((_, reject) => {
    timer = setTimeout(() => {
      const reason = new TimeoutReason(RESET_OPERATION_TIMEOUT_MS);
      reject(new ResetCancelledError('timeout'));
      operationCtrl.abort(reason);
    }, RESET_OPERATION_TIMEOUT_MS);

    // External cancel path: user-fired controller.abort() must also reject.
    if (merged.aborted) {
      reject(classifyAbort(merged.reason));
    } else {
      merged.addEventListener('abort', () => {
        reject(classifyAbort(merged.reason));
      }, { once: true });
    }
  });

  return await Promise.race([summaryPromise, racePromise]);
} finally {
  if (timer) clearTimeout(timer);
  handler.setCurrentResetController(threadId, null);
  handler.setResetInFlight(threadId, false);  // ← MOVED from line 242 into outer finally
}

// helper, module-private in session-reset.ts (NOT abort-utils.ts —
// helper carries ResetCancelledError knowledge; placing it in
// abort-utils would invert the dependency direction):
function classifyAbort(reason: unknown): Error {
  if (reason instanceof TimeoutReason) return new ResetCancelledError('timeout');
  if (reason instanceof ResetCancelledError) return reason; // user-fired with explicit reason
  return new ResetCancelledError('user');
}
```

**`setResetInFlight(false)` move**: v0.5.3-and-prior had it at line 242 inner finally. v0.5.4 needs it AFTER `clearTimeout` and AFTER `setCurrentResetController(null)` to prevent ordering bugs where a second `/councilcancel` arriving in the cleanup window finds stale state. Single outer finally is the correct reorder.

## 5. Cancel lifecycle state machine

```
            ┌─────────┐
            │  idle   │
            └────┬────┘
                 │ user fires /councilreset
                 │ guards pass
                 │ resetInFlight = true
                 │ currentResetController = new AbortController()
                 ▼
         ┌─────────────────┐
         │   in-flight     │
         │ - LLM call      │
         │ - DB recordSnap │
         │ - seal segment  │
         │ - open segment  │
         └─┬───────┬───────┘
           │       │
   summary │       │ abort fires
   resolves│       │ (timeout OR /councilcancel)
   AND seal│       │
   AND open│       │
   succeed │       │
           │       │
           ▼       ▼
       settled   cancelled-or-timed-out
       (return)   (throw ResetCancelledError)
                  ↓
                  R1/R2/R3/R4 rollback per §6
                  ↓
       finally: clearTimeout, controller=null, resetInFlight=false
```

## 6. Race scenarios

**Load-bearing JS guarantee** (codex round-1 should verify): `Promise.race([summaryPromise, racePromise])` cannot inject the race-reject between sync statements inside `summaryPromise`'s body. Once `summaryPromise`'s async function resumes from an `await` and enters a synchronous block, that block runs to completion (or the next `await`) before any other microtask — including the race-reject — is dispatched. This is what makes R2/R3 below collapse into R4.

| ID | Cancel arrives... | Action | Result |
|----|-------------------|--------|--------|
| R0 | Before `reset()` is called, signal already aborted | `merged.aborted === true` at the `if (merged.aborted)` early-check (§4.1). Reject synchronously before timer install. | Throw `ResetCancelledError(reason inferred from merged.reason)`. Guards have already passed; `resetInFlight` is `true` and was just set, finally clears it. No LLM call, no DB write. |
| R1 | Before `respondDeterministic` returns (LLM in-flight) | merged signal aborts → SDK aborts (Claude/OpenAI/Custom) OR Promise.race rejects (Google, see v0.5.3 §0.2 option C) | Throw `ResetCancelledError`. Nothing in DB, segment intact. `resetInFlight=false` via finally. |
| R2 | After `db.recordSnapshot` (line 208), before `sealCurrentSegment` (line 213) | Race promise wins; summaryPromise's next `await` (none — line 213 is sync) means summaryPromise actually proceeds to seal+open. **This race is impossible** — there is no `await` between line 208 and line 233. Cancel that arrives during this synchronous block has its callback queued and runs after open completes. R2 collapses into R4. | (impossible — see R4) |
| R3 | After `sealCurrentSegment` (line 213), before `openNewSegment` (line 223) | Same as R2 — no `await` between 213 and 223. Cancel callback runs after R4. | (impossible — see R4) |
| R4 | After `openNewSegment` (line 223) returns, before `Promise.race` settles | summaryPromise has already returned. Race-reject's `setTimeout` callback eventually runs, finds promise already settled, `reject` is no-op on already-settled Promise.race winner. | Reset succeeds normally. Cancel was "too late". User sees success message. |
| R5 | During microtask gap between `respondDeterministic` resolution (line 161) and `validateResetSummaryMarkdown` (line 178) | Validation runs synchronously after `await` resolves. AbortController.abort() takes effect at next `await`, of which there is none until DB write. If summary IS valid, R5 collapses into the impossible-because-synchronous R2/R3/R4. If summary is INVALID, `MalformedResetSummaryError` throws first; cancel never observed. | Validation error wins (matches §2.5). |
| R6 | `/councilcancel` fired with no reset in progress | Adapter reads `getCurrentResetController(threadId)`, gets `null`, returns user-visible "No reset in progress." | No state change. |
| R7 | Two `/councilcancel` fire concurrently | First call's `controller.abort()` settles the controller. Second call reads `getCurrentResetController(threadId)`. If finally hasn't run yet, gets the same controller — calls `.abort()` again, JS spec no-op. If finally already cleared, gets `null` (R6). | Either branch is safe. |

**The R2/R3 collapse is load-bearing**: cancel cannot arrive between line 208 and line 233 because that block is synchronous (no `await`). This means we DO NOT need cancel-aware rollback inside seal/open. The existing `safeDeleteSnapshot` (line 249) covers seal-fail / open-fail, and cancel-after-LLM is exclusively R1 (no DB write yet) or R4 (everything done, success).

## 7. Test plan

### 7.1 Unit: `AgentWorker.respondDeterministic` forwards signal into provider.chat

`respondDeterministic` does not pre-check signal — it forwards into `provider.chat({ signal })` and lets the SDK reject. This matches v0.5.3 `respond()` behavior; pre-checking would diverge for no benefit.

```ts
test('respondDeterministic forwards signal into provider.chat', async () => {
  const controller = new AbortController();
  controller.abort();
  const fakeProvider = { chat: vi.fn().mockRejectedValue(new Error('AbortError')) };
  const worker = new AgentWorker(/*...*/, fakeProvider);
  await expect(
    worker.respondDeterministic(messages, 'synthesizer', controller.signal),
  ).rejects.toThrow();
  expect(fakeProvider.chat).toHaveBeenCalledTimes(1);
  expect(fakeProvider.chat.mock.calls[0][1].signal).toBe(controller.signal);
});
```

### 7.2 Integration: `/councilcancel` mid-LLM (R1)

```ts
test('/councilcancel during LLM call throws ResetCancelledError reason=user', async () => {
```
