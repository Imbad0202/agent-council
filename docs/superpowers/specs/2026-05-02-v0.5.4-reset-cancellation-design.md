## 2. Invariants

### 2.1 `resetInFlight` always cleared

Existing `try { ... } finally { handler.setResetInFlight(threadId, false); }` in `session-reset.ts:117/242` is the single unwind point. Cancel paths throw `ResetCancelledError` from inside `try`, hit the same `finally`. **No new flag-clearing site needed.** Spec adds NO additional `setResetInFlight(false)` calls; auditing this prevents double-clear / use-after-clear regressions.

### 2.2 Cancel does NOT trigger snapshot rollback

**Round-1 P2-1 correction.** R2/R3 collapse into R4 (codex-verified — lines 208–233 are synchronous, race rejection cannot preempt the sync block). This means cancel arriving after `db.recordSnapshot` is "too late" — the reset succeeds and the user sees the success path. There is NO cancel-driven `safeDeleteSnapshot` call.

`safeDeleteSnapshot` (`session-reset.ts:249`) ONLY runs when `sealCurrentSegment` (line 213) or `openNewSegment` (line 223) themselves throw. That rollback is independent of cancellation and unchanged from v0.5.3-and-prior.

The cancellation invariant is therefore: **cancel either prevents the snapshot row from being written (R0/R1) or arrives too late and the reset commits (R4). Cancel never partially commits.** This is what makes the post-LLM abort gate (§4.1, P1-1 fix) the load-bearing invariant — it ensures the "cancel either prevents or commits, never partially commits" property by refusing to write the snapshot when cancel landed during the LLM await.

### 2.3 Second cancel is no-op, not error

If user fires `/councilcancel` twice (or fires it while reset is already settling), the second `controller.abort()` on an already-settled `AbortController` is a JS-spec no-op. `currentResetController` clear-on-finally (§3.3) means the second `/councilcancel` finds `null` and returns the friendly "no reset in progress" message (R6).

### 2.4 Thread isolation

`currentResetController` lives on `SessionState[threadId]`, NOT on `SessionReset` instance. A `/councilcancel` issued in thread A reads thread A's controller only — thread B reset is unaffected. Telegram chat_id and CLI thread-id naming match existing `setResetInFlight(threadId, ...)` invariant.

### 2.5 Order at the post-await microtask edge

After `await this.facilitator.respondDeterministic(...)` resolves, the next synchronous statements are: (1) **R1' abort gate** (`if (merged.aborted) throw classifyAbort(...)` — round-1 P1-1 fix), then (2) `validateResetSummaryMarkdown`. Both run synchronously after the `await` — no further `await` between them, so neither can be preempted by cancel-via-AbortController.

**Order at this edge**: R1' gate fires FIRST. If `merged.aborted` is true at the gate, the cancel error throws and validation never runs. If `merged.aborted` is false, validation runs; it either passes (proceed to sync block 208–233) or throws `MalformedResetSummaryError`.

This means: if cancel landed during the LLM await, R1' wins over a malformed summary error — the user sees `ResetCancelledError`, not `MalformedResetSummaryError`. This is intentional. A cancelled reset that also happened to receive a malformed summary should surface as cancellation; the malformed-summary signal is moot once the user has cancelled. Round-0 spec said "validation error wins" — round-1 corrects this to "cancel wins, validation second" because the gate's purpose is to prevent ANY commit-or-error path that the user already abandoned.

Lifecycle (round-1 P2-2 — single canonical ordering, see §4.1 for the matching code):

**Setup (inside outer `try`, after `setResetInFlight(true)`)**:
1. `new AbortController()` → `operationCtrl`
2. `mergeSignals(options?.signal, operationCtrl.signal)` → `merged`
3. **P1-2 gate**: if `merged.aborted`, throw `classifyAbort(merged.reason)` BEFORE constructing summaryPromise. Outer finally clears `resetInFlight`.
4. `handler.setCurrentResetController(threadId, operationCtrl)` (inside inner try)
5. Install `setTimeout` timer + race
6. `await Promise.race([summaryPromise, racePromise])`

**Cleanup (innermost finally first, then outer)**:
1. Inner finally: `clearTimeout(timer)` + `handler.setCurrentResetController(threadId, null)`
2. Outer finally: `handler.setResetInFlight(threadId, false)`

This ordering makes a second `/councilcancel` arriving during cleanup safe (R7): it sees `null` controller (cleared in inner finally) before `resetInFlight=false` (cleared in outer finally). The opposite order would let a second `/councilcancel` find `resetInFlight=false` AND a non-null controller — visible-but-unowned state.

**Adapter contract** (round-1 P2-3, round-2 P2-r2-1 corrected): `/councilcancel` handlers reach the controller via existing `SessionResetWiring.deliberationHandler`. NO new wiring channel; `setSessionResetWiring` already passes the handler reference when a facilitator is configured.

Telegram `bot.ts:427` registers reset-family commands inside `if (sessionReset)` — the wiring object is truthy in both DB-only and full configurations, so the `if` does NOT discriminate facilitator presence.

### 3.4 `ResetCancelledError`

```ts
// src/council/session-reset-errors.ts (new export)
export type ResetCancelReason = 'user' | 'timeout';

export class ResetCancelledError extends Error {
  constructor(public readonly reason: ResetCancelReason) {
    super(`Reset cancelled (reason: ${reason})`);
    this.name = 'ResetCancelledError';
  }
}
```

Adapter pattern (mirrors existing `BlindReviewActiveError` / `ResetInProgressError` `instanceof` checks):

```ts
if (err instanceof ResetCancelledError) {
  if (err.reason === 'user') {
    reply('Reset cancelled.');
  } else {
    reply('Reset timed out (no facilitator response within 30s). Try again.');
  }
}
```

**SDK error class wrapping (round-1 P1-3, round-2 P1-r2-1 broadened)**: provider SDKs may reject in-flight chats with `APIUserAbortError` (Anthropic / OpenAI), `AbortError` (`AbortController` standard), or `code: 'ABORT_ERR'` variants. Without reclassification, an adapter doing `instanceof ResetCancelledError` would NOT match — the caller would see the raw SDK class.

The §4.1 race try/catch wrapper calls `classifyAbort(merged.reason)` whenever `merged.aborted` is true at catch time, **regardless of the err shape**. Round-2 widened the condition from `isAbortError(err) && merged.aborted` to `merged.aborted` alone, because round-1's narrower condition let non-abort errors (e.g. a `MalformedResetSummaryError` thrown after cancel landed) leak through and contradict §2.5's "cancel wins" guarantee. The widened condition: any throw exiting `Promise.race` while the operation is aborted is treated as a cancellation, including raw SDK abort classes AND any other error class. Non-aborted throws (genuine validation failure on a non-cancelled reset) propagate unchanged.

### 3.5 Telegram concurrent dispatch via `@grammyjs/runner` (round-4 P1-r4-1, K1)

**Why this is required**: `src/adapters/telegram.ts:127` calls `await listenerBot.start({ drop_pending_updates: true, onStart: ... })`. Default grammY `bot.start()` handles updates sequentially per `node_modules/grammy/out/bot.js:188` (`// handle updates sequentially (!)`). Sequential dispatch means while `/councilreset` handler is awaiting `wiring.reset.reset(...)` (could be 30s in worst case), grammY does not pull or dispatch the next update. `/councilcancel` issued mid-reset queues behind reset and only fires after reset returns — by which time the cancellation is moot.

**Codex round-4 verified** the sequential behavior in installed grammy 1.35.0 source. Without K1 the entire B2 user-cancel surface is non-functional in Telegram.

**Resolution**: adopt `@grammyjs/runner`, grammY's official concurrent dispatcher (`grammy.dev/plugins/runner`).

**Dependency change** (`package.json`):
```diff
   "dependencies": {
     ...
     "grammy": "^1.35.0",
+    "@grammyjs/runner": "^2.0.0",
     ...
   }
```

(Use whatever `@grammyjs/runner` major version is current at implementation time — `^2.0.0` is the current line as of 2026-05-02. Implementation should run `npm install @grammyjs/runner` and pin to whatever resolves.)

**Adapter change** (`src/adapters/telegram.ts:127`): replace `bot.start(...)` with `run(bot, ...)`:
```diff
- await listenerBot.start({ drop_pending_updates: true, onStart: () => console.log('Telegram adapter running.') });
+ const runner = run(listenerBot, {
+   runner: { drop_pending_updates: true },
+ });
+ console.log('Telegram adapter running (concurrent dispatch).');
+ // Save runner handle for shutdown — required so `runner.stop()` can
+ // be called from the adapter's stop() method (parallels former bot.stop()).
+ this.runner = runner;
```

**Shutdown semantics**: `bot.stop()` (if called anywhere) is replaced by `runner.stop()`. Adapter's `stop()` method (if it exists) must be updated to await `this.runner.stop()`. Verify no other code path assumes `listenerBot.start()`'s synchronous-loop semantics.

**Pre-start conflict-retry logic preserved** (lines 109-126 in adapter): the `getUpdates({ offset: -1 })` 6-attempt slot-acquisition loop runs BEFORE bot startup, unrelated to dispatch concurrency. Keep verbatim.

**Concurrency model implications**:
- `/councilcancel` while `/councilreset` is in-flight: both handlers run concurrently; cancel handler reads `getCurrentResetController(threadId)` and calls `.abort()`, which propagates through `merged` signal into the in-flight reset and triggers the §4.1 race wrapper.
- Two `/councilreset` from same thread concurrently: existing `setResetInFlight(threadId, true)` guard at `session-reset.ts:116` already protects via `ResetInProgressError`. Concurrency does NOT bypass it because the guard is per-thread state.
- Two `/councilcancel` close together: covered by R7. JS-spec no-op on already-settled controller.

**What this does NOT enable** (round-4 scope guard):
- Concurrent `runDeliberation` calls — those are gated by `setDeliberationInFlight` on the deliberation side, not affected here.
- Multi-thread Telegram bot processing — `groupChatId` filter still scopes to one chat.
- This is purely about preventing handler-level head-of-line blocking inside the adapter's update dispatch.

**Test impact**: §7.6 Telegram cancel integration test was untestable in round-3 (mock bot.start only dispatches one handler at a time). With runner, the test becomes a real integration: spawn reset handler, then cancel handler concurrently, assert reset rejects with `ResetCancelledError`. See §7.6 round-4 update.

### 4.1 detail — the race+abort pattern (round-2 revision)

This pattern closes round-1 P1-1 (background commit), P1-2 (R0 false), P1-3 (SDK abort wrapping), P2-2 (lifecycle ordering), round-2 P1-r2-1 (catch over-narrow), and round-3 P1-r3-1 (race wrapper extracted as testable seam). Annotations show which finding each block addresses.

```ts
// session-reset.ts, near top of file:
const RESET_OPERATION_TIMEOUT_MS = 30_000;

// helper — module-private (round-0 decision, kept in round-1):
function classifyAbort(reason: unknown): Error {
  if (reason instanceof TimeoutReason) return new ResetCancelledError('timeout');
  if (reason instanceof ResetCancelledError) return reason; // user-fired with explicit reason
  return new ResetCancelledError('user');
}

// [round-3 P1-r3-1] Race wrapper extracted as a module-EXPORTED named
// function so §7.7a/b/c unit tests exercise THIS code, not a duplicated
// catch block inside the test. Without this extraction, removing or
// breaking the catch inside SessionReset.reset() would not be caught
// by §7.7 tests because they would still test the locally-rewritten
// catch in the test file. Seam = test target.
//
// Exported (not module-private) so tests in tests/council/ can import
// without abusing internal access. Export keeps this seam stable across
// future refactors — if the catch logic changes, this signature must
// change too, forcing test updates.
export async function awaitResetRace<T>(
  summaryPromise: Promise<T>,
  racePromise: Promise<never>,
  signal: AbortSignal,
): Promise<T> {
  try {
    return await Promise.race([summaryPromise, racePromise]);
  } catch (err) {
    if (signal.aborted) {
      throw classifyAbort(signal.reason);
    }
    throw err;
  }
}

// inside reset(), after all existing guards (line 78–114):
handler.setResetInFlight(threadId, true);
try {
  const operationCtrl = new AbortController();
  const merged = mergeSignals(options?.signal, operationCtrl.signal)!;
  // merged is non-null because operationCtrl.signal is always present.

  // [P1-2 fix] Pre-check BEFORE constructing summaryPromise.
  // Async function bodies run synchronously up to the first await, so a
  // summaryPromise constructed before this check would call provider.chat
  // (line 172 of agent-worker.ts) before the race could reject.
  if (merged.aborted) {
    throw classifyAbort(merged.reason);
  }

  handler.setCurrentResetController(threadId, operationCtrl);
  let timer: NodeJS.Timeout | undefined;
  try {
    // summaryPromise body — same as existing lines 118–240, except:
    //   (a) respondDeterministic forwards `merged` (signal forwarding)
    //   (b) [P1-1 fix] post-await abort gate before validation/persist
    const summaryPromise = (async () => {
      const messages = handler.getCurrentSegmentMessages(threadId);
      const existing = effectiveResetSnapshots(threadId, this.db, this.artifactDb);
      // ... build priorSummariesMsg, summaryMsg, facilitatorMessages
      //     (same as current lines 118–160)

      const response = await this.facilitator.respondDeterministic(
        facilitatorMessages,
        'synthesizer',
        merged,                    // ← signal forwarding (§3.1)
      );

      // [P1-1 fix] If cancel/timeout landed while LLM was in flight,
      // refuse to commit even if the SDK happened to return a complete
      // response before honoring the abort. Without this gate, a slow-to-
      // honor SDK can let the cancelled reset commit DB+segment state
      // AFTER cleanup ran — a background-write bug. Codex round-1 P1-1.
      if (merged.aborted) {
        throw classifyAbort(merged.reason);
      }

      const summaryMarkdown = response.content;
      const validation = validateResetSummaryMarkdown(summaryMarkdown);
      if (!validation.valid) {
        throw new MalformedResetSummaryError(validation.missingSections);
      }
      // ... parseSummaryMetadata, computeNextSegmentIndex, build snapshot
      //     (same as current lines 183–206)

      this.db.recordSnapshot(snapshot);

      // Synchronous block 208–233 (codex round-1 verified): no await between
      // recordSnapshot and openNewSegment, so race rejection cannot preempt.
      // Existing seal/open rollback via safeDeleteSnapshot is unchanged.
      try {
        handler.sealCurrentSegment(threadId, snapshotId);
      } catch (sealErr) {
        this.safeDeleteSnapshot(snapshotId, sealErr);
        throw sealErr;
      }
      try {
        handler.openNewSegment(threadId);
      } catch (openErr) {
        try { handler.unsealCurrentSegment(threadId); } catch {}
        this.safeDeleteSnapshot(snapshotId, openErr);
        throw openErr;
      }

      return { snapshotId, summaryMarkdown, metadata: snapshot.metadata, segmentIndex };
    })();

    const racePromise = new Promise<never>((_, reject) => {
      timer = setTimeout(() => {
        // Single timer drives BOTH abort and reject (v0.5.3 §5.1 site 5).
        // Reject synchronously before abort so race winner is deterministic.
        reject(new ResetCancelledError('timeout'));
        operationCtrl.abort(new TimeoutReason(RESET_OPERATION_TIMEOUT_MS));
      }, RESET_OPERATION_TIMEOUT_MS);

      // External cancel path: user-fired controller.abort() must also reject.
      // (merged.aborted at function entry is handled by the §4.1 P1-2 gate
      // above — by here, merged is not yet aborted.)
      merged.addEventListener('abort', () => {
        reject(classifyAbort(merged.reason));
      }, { once: true });
    });

    // [P1-3 + round-2 P1-r2-1 + round-3 P1-r3-1 fix] Race-await delegated
    // to awaitResetRace helper.
    // Two failure modes converge in the helper's catch:
    //   (a) Provider SDK rejects summaryPromise with APIUserAbortError /
    //       AbortError before racePromise rejects (SDK abort listener fires
    //       faster than our addEventListener) — round-1 P1-3.
    //   (b) summaryPromise rejects with a NON-abort error AND merged.aborted
    //       is true (e.g. provider non-abort rejection during await with
    //       cancel landing concurrently) — round-2 P1-r2-1.
    // §2.5 "cancel wins" requires both modes to surface ResetCancelledError
    // to the caller. The helper tests `signal.aborted` alone, NOT err shape:
    // any throw exiting Promise.race while the operation is aborted gets
    // reclassified. Non-aborted-state throws propagate unchanged.
    //
    // [round-3 P1-r3-1] Helper extracted so §7.7 tests can import and
    // exercise THIS code path directly. Inline catch (round-1/round-2) was
    // not testable as a regression seam — removing the catch would not
    // make round-2 §7.7 tests fail.
    return await awaitResetRace(summaryPromise, racePromise, merged);
  } finally {
    // Inner cleanup: clear timer FIRST (so any late-firing setTimeout
    // callback doesn't run after controller is null), then clear controller.
    if (timer) clearTimeout(timer);
    handler.setCurrentResetController(threadId, null);
  }
} finally {
  // Outer cleanup: clear flag last so a second /councilcancel arriving
  // during cleanup never sees `resetInFlight=false` AND a non-null
  // controller simultaneously (round-1 P2-2 ordering).
  handler.setResetInFlight(threadId, false);
}
```

**Lifecycle ordering invariant (round-1 P2-2)**:
- During reset: `resetInFlight=true` AND `currentResetController=<controller>`
- After cleanup: `resetInFlight=false` AND `currentResetController=null`
- Cleanup transition: controller cleared FIRST (inner finally), flag cleared LAST (outer finally). A second `/councilcancel` reading `resetInFlight=true` is guaranteed to see a non-null controller; reading `resetInFlight=false` is guaranteed to see `null` controller.

**Why two nested try/finally instead of one**: round-0 used a single outer finally; round-1 codex P2-2 noted that `setCurrentResetController(controller)` placed BEFORE the try would leak `resetInFlight=true` if `mergeSignals` threw at setup. Two finallys make setup-throw rollback automatic: the P1-2 early-throw, the `mergeSignals` setup throw, and any other setup error all go through outer finally and clear the flag.

## 5. Cancel lifecycle state machine

```
            ┌─────────┐
            │  idle   │
            └────┬────┘
                 │ user fires /councilreset
                 │ guards pass
                 │ resetInFlight = true
                 │ currentResetController = new AbortController()
                 ▼
         ┌─────────────────┐
         │   in-flight     │
         │ - LLM call      │
         │ - DB recordSnap │
         │ - seal segment  │
         │ - open segment  │
         └─┬───────┬───────┘
           │       │
   summary │       │ abort fires
   resolves│       │ (timeout OR /councilcancel)
   AND seal│       │
   AND open│       │
   succeed │       │
           │       │
           ▼       ▼
       settled   cancelled-or-timed-out
       (return)   (throw ResetCancelledError)
                  ↓
                  R1/R2/R3/R4 rollback per §6
                  ↓
       finally: clearTimeout, controller=null, resetInFlight=false
```

## 6. Race scenarios

**Load-bearing JS guarantee** (codex round-1 should verify): `Promise.race([summaryPromise, racePromise])` cannot inject the race-reject between sync statements inside `summaryPromise`'s body. Once `summaryPromise`'s async function resumes from an `await` and enters a synchronous block, that block runs to completion (or the next `await`) before any other microtask — including the race-reject — is dispatched. This is what makes R2/R3 below collapse into R4.

| ID | Cancel arrives... | Action | Result |
|----|-------------------|--------|--------|
| R0 | `options.signal` already aborted at function entry | The `if (merged.aborted)` check at §4.1 (BEFORE `setCurrentResetController` and BEFORE `summaryPromise` construction, round-1 P1-2 fix) throws `classifyAbort(merged.reason)` synchronously. Outer finally clears `resetInFlight`. | Throw `ResetCancelledError`. `setResetInFlight(true)` happened, but **no LLM call, no DB write, no controller registered**. Guards already passed when the early-check fires. |
| R1 | After §4.1 early-check passes; before `respondDeterministic` returns (LLM in-flight) | `merged.addEventListener('abort')` listener calls `reject(classifyAbort(merged.reason))` on racePromise. ALSO: provider SDK abort listener may reject summaryPromise with raw SDK error first. The §4.1 try/catch wrapper reclassifies (round-1 P1-3 fix). | Throw `ResetCancelledError`. Nothing in DB, segment intact. `resetInFlight=false` via outer finally. |
| R1' | `respondDeterministic` resolved with a complete response, but `merged.aborted` is true (cancel landed during the LLM await but SDK ignored / honored too slowly) | Post-await abort gate inside summaryPromise body (round-1 P1-1 fix) throws `classifyAbort(merged.reason)`. | Throw `ResetCancelledError`. **No DB write, no segment mutation** even though the LLM returned content. This is the round-1 P1-1 critical fix — without this gate, summaryPromise would proceed to commit in the background after cleanup ran. |
| R2 | After `db.recordSnapshot` (line 208), before `sealCurrentSegment` (line 213) | No `await` between 208 and 233 (codex round-1 verified). Async function bodies run synchronously across this block; race rejection cannot preempt sync code in summaryPromise. Cancel callback queued, runs AFTER open completes. R2 collapses into R4. | (impossible — see R4) |
| R3 | After `sealCurrentSegment` (line 213), before `openNewSegment` (line 223) | Same as R2 — no `await` between 213 and 223. R3 collapses into R4. | (impossible — see R4) |
| R4 | After `openNewSegment` returns, before `Promise.race` settles | summaryPromise has already returned its result. Race-reject's setTimeout/abort-listener fires later, but Promise.race winner is already determined. | Reset succeeds. Cancel was "too late". User sees success message. R4 is the "cancel arriving after commit" non-event. |
| R5 | Microtask gap between `respondDeterministic` resolution and `validateResetSummaryMarkdown` | Validation runs synchronously after the `await` resolves. The R1' post-await abort gate also runs synchronously in the same window. Order: gate FIRST (line just after `await`), then validate. If `merged.aborted` already, R1' fires. If not aborted yet, validation runs and either passes (proceed to sync block) or throws `MalformedResetSummaryError`. | If aborted: R1'. Else if invalid summary: validation error wins. Else: success path. |
| R6 | `/councilcancel` fired with no reset in progress | Adapter reads `getCurrentResetController(threadId)`, gets `null`, replies "No reset in progress." | No state change. |
| R7 | Two `/councilcancel` fire concurrently | First call's `controller.abort()` settles the controller. Second call reads `getCurrentResetController(threadId)`. (a) If inner finally hasn't run yet, gets the same controller — `.abort()` is a JS-spec no-op on an already-settled controller. (b) If inner finally already cleared, gets `null` (R6). The lifecycle ordering invariant (§3.3) guarantees a second cancel never sees `resetInFlight=false` AND a non-null controller. | Either branch is safe. |
| R8 | SDK abort listener rejects `summaryPromise` with `APIUserAbortError` / `AbortError` (or any other error class — round-2 broadening) BEFORE racePromise's `reject` runs | The §4.1 race try/catch wrapper (extracted as `awaitResetRace` helper in round-3) detects `signal.aborted` alone — regardless of err shape — and reclassifies via `classifyAbort(signal.reason)`. Round-1 narrow `isAbortError(err) && merged.aborted` was broadened in round-2 P1-r2-1 to handle non-abort error classes that escape concurrent with cancellation. | Throw `ResetCancelledError` to caller, NOT raw SDK error class or other thrown class. Adapters' `instanceof ResetCancelledError` checks always match when cancel landed during the race. |

**The R2/R3 collapse is load-bearing** (codex round-1 verified): cancel cannot arrive between line 208 and line 233 because that block is synchronous (no `await`), AND `Promise.race` cannot preempt sync code inside summaryPromise's body. This means we DO NOT need cancel-aware rollback inside seal/open. The existing `safeDeleteSnapshot` (line 249) covers seal-fail / open-fail, and cancel-after-LLM is exclusively R1 (LLM in-flight, no commit), R1' (LLM done but cancel landed first, post-await gate refuses to commit), or R4 (synchronous block already ran, success).

**R1' is the round-1 P1-1 critical fix**: without the post-await abort gate, the SDK could return a complete response after cancel landed, and summaryPromise would proceed to write DB / mutate segment in the background AFTER outer cleanup ran. The gate enforces the cancellation invariant (§2.2): cancel either prevents the snapshot row from being written, or arrives too late and reset commits. Cancel never partially commits.

### 7.5 Integration: thread isolation (§2.4)

Round-3 P2-r3-3 cleanup discipline: round-2 fix called `vi.useRealTimers()` only on the happy path, so a mid-test failure left fake timers leaking into adjacent tests. Round-3 wraps the whole body in `try/finally` (matching `tests/council/facilitator-abort.test.ts:34-45` and `tests/council/artifact-invoke.test.ts:39-42` repo convention) and attaches rejection handlers to `promiseB` BEFORE cancellation triggers, so vitest never sees an unhandled rejection regardless of which assertion fails.

```ts
test('/councilcancel on thread A does not abort thread B reset', async () => {
  vi.useFakeTimers();
  try {
    const promiseA = reset.reset(handler, 1); // hangs
    const promiseB = reset.reset(handler, 2); // hangs

    // [P2-r3-3] Attach rejection handler to promiseB EARLY so any later
    // failure (assertion throw, fake-timer mishap) doesn't surface as
    // unhandled rejection. The handler is a no-op until awaited later.
    let bSettled = false;
    let bError: unknown;
    promiseB.catch((err) => { bSettled = true; bError = err; });

    // Cancel A only.
    await cliRouter.handle('/councilcancel', { threadId: 1 });
    await expect(promiseA).rejects.toMatchObject({ reason: 'user' });

    // promiseB still pending — verify by checking the early-attached flag.
    // Advance only enough for A-driven microtasks to flush, without
    // crossing the 30s timeout boundary.
    await vi.advanceTimersByTimeAsync(100);
    await Promise.resolve(); // microtask flush
    expect(bSettled).toBe(false);

    // Test cleanup: cancel B explicitly. Without this, fake timers' 30s
    // advance at test teardown (or test-pollution into the next test)
    // would surface an unhandled rejection.
    await cliRouter.handle('/councilcancel', { threadId: 2 });
    await expect(promiseB).rejects.toMatchObject({ reason: 'user' });
    expect(bError).toMatchObject({ reason: 'user' });
  } finally {
    vi.useRealTimers();
  }
});
```
