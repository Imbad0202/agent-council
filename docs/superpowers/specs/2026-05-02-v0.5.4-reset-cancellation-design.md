## 2. Invariants

### 2.1 `resetInFlight` always cleared

Existing `try { ... } finally { handler.setResetInFlight(threadId, false); }` in `session-reset.ts:117/242` is the single unwind point. Cancel paths throw `ResetCancelledError` from inside `try`, hit the same `finally`. **No new flag-clearing site needed.** Spec adds NO additional `setResetInFlight(false)` calls; auditing this prevents double-clear / use-after-clear regressions.

### 2.2 Cancel does NOT trigger snapshot rollback

**Round-1 P2-1 correction.** R2/R3 collapse into R4 (codex-verified — lines 208–233 are synchronous, race rejection cannot preempt the sync block). This means cancel arriving after `db.recordSnapshot` is "too late" — the reset succeeds and the user sees the success path. There is NO cancel-driven `safeDeleteSnapshot` call.

`safeDeleteSnapshot` (`session-reset.ts:249`) ONLY runs when `sealCurrentSegment` (line 213) or `openNewSegment` (line 223) themselves throw. That rollback is independent of cancellation and unchanged from v0.5.3-and-prior.

The cancellation invariant is therefore: **cancel either prevents the snapshot row from being written (R0/R1) or arrives too late and the reset commits (R4). Cancel never partially commits.** This is what makes the post-LLM abort gate (§4.1, P1-1 fix) the load-bearing invariant — it ensures the "cancel either prevents or commits, never partially commits" property by refusing to write the snapshot when cancel landed during the LLM await.

### 2.3 Second cancel is no-op, not error

If user fires `/councilcancel` twice (or fires it while reset is already settling), the second `controller.abort()` on an already-settled `AbortController` is a JS-spec no-op. `currentResetController` clear-on-finally (§3.3) means the second `/councilcancel` finds `null` and returns the friendly "no reset in progress" message (R6).

### 2.4 Thread isolation

`currentResetController` lives on `SessionState[threadId]`, NOT on `SessionReset` instance. A `/councilcancel` issued in thread A reads thread A's controller only — thread B reset is unaffected. Telegram chat_id and CLI thread-id naming match existing `setResetInFlight(threadId, ...)` invariant.

### 2.5 Order at the post-await microtask edge

After `await this.facilitator.respondDeterministic(...)` resolves, the next synchronous statements are: (1) **R1' abort gate** (`if (merged.aborted) throw classifyAbort(...)` — round-1 P1-1 fix), then (2) `validateResetSummaryMarkdown`. Both run synchronously after the `await` — no further `await` between them, so neither can be preempted by cancel-via-AbortController.

**Order at this edge**: R1' gate fires FIRST. If `merged.aborted` is true at the gate, the cancel error throws and validation never runs. If `merged.aborted` is false, validation runs; it either passes (proceed to sync block 208–233) or throws `MalformedResetSummaryError`.

This means: if cancel landed during the LLM await, R1' wins over a malformed summary error — the user sees `ResetCancelledError`, not `MalformedResetSummaryError`. This is intentional. A cancelled reset that also happened to receive a malformed summary should surface as cancellation; the malformed-summary signal is moot once the user has cancelled. Round-0 spec said "validation error wins" — round-1 corrects this to "cancel wins, validation second" because the gate's purpose is to prevent ANY commit-or-error path that the user already abandoned.

### 3.3 `SessionState` per-thread controller

`HandlerForReset` interface gains:

```ts
getCurrentResetController(threadId: number): AbortController | null;
setCurrentResetController(threadId: number, controller: AbortController | null): void;
```

Implementation on `DeliberationHandler` private `SessionState`:

```ts
interface SessionState {
  // ...existing fields
  currentResetController?: AbortController;
}
```

Lifecycle (round-1 P2-2 — single canonical ordering, see §4.1 for the matching code):

**Setup (inside outer `try`, after `setResetInFlight(true)`)**:
1. `new AbortController()` → `operationCtrl`
2. `mergeSignals(options?.signal, operationCtrl.signal)` → `merged`
3. **P1-2 gate**: if `merged.aborted`, throw `classifyAbort(merged.reason)` BEFORE constructing summaryPromise. Outer finally clears `resetInFlight`.
4. `handler.setCurrentResetController(threadId, operationCtrl)` (inside inner try)
5. Install `setTimeout` timer + race
6. `await Promise.race([summaryPromise, racePromise])`

**Cleanup (innermost finally first, then outer)**:
1. Inner finally: `clearTimeout(timer)` + `handler.setCurrentResetController(threadId, null)`
2. Outer finally: `handler.setResetInFlight(threadId, false)`

This ordering makes a second `/councilcancel` arriving during cleanup safe (R7): it sees `null` controller (cleared in inner finally) before `resetInFlight=false` (cleared in outer finally). The opposite order would let a second `/councilcancel` find `resetInFlight=false` AND a non-null controller — visible-but-unowned state.

**Adapter contract** (round-1 P2-3): `/councilcancel` handlers reach the controller via existing `SessionResetWiring.deliberationHandler`. NO new wiring channel; `setSessionResetWiring` already passes the handler reference. Adapters call `wiring.deliberationHandler.getCurrentResetController(threadId)` and invoke `.abort(new ResetCancelledError('user'))` if non-null. **Telegram-with-DB-only-no-facilitator wiring**: `/councilcancel` is registered but `getCurrentResetController` always returns `null` (because `SessionReset.reset` cannot run without a facilitator — `SessionResetWiring` is `null` in that case, see `index.ts:225-230`). Adapter shows "No reset in progress." and never calls abort.

### 3.4 `ResetCancelledError`

```ts
// src/council/session-reset-errors.ts (new export)
export type ResetCancelReason = 'user' | 'timeout';

export class ResetCancelledError extends Error {
  constructor(public readonly reason: ResetCancelReason) {
    super(`Reset cancelled (reason: ${reason})`);
    this.name = 'ResetCancelledError';
  }
}
```

Adapter pattern (mirrors existing `BlindReviewActiveError` / `ResetInProgressError` `instanceof` checks):

```ts
if (err instanceof ResetCancelledError) {
  if (err.reason === 'user') {
    reply('Reset cancelled.');
  } else {
    reply('Reset timed out (no facilitator response within 30s). Try again.');
  }
}
```

**SDK error class wrapping (round-1 P1-3)**: provider SDKs may reject in-flight chats with `APIUserAbortError` (Anthropic / OpenAI), `AbortError` (`AbortController` standard), or `code: 'ABORT_ERR'` variants. Without reclassification, an adapter doing `instanceof ResetCancelledError` would NOT match — the caller would see the raw SDK class. The fix lives in the §4.1 race try/catch wrapper, which calls `classifyAbort(merged.reason)` whenever `isAbortError(err) && merged.aborted`. Adapters therefore always see `ResetCancelledError`, never the SDK's raw abort class.

### 4.1 detail — the race+abort pattern (round-1 revision)

This pattern closes round-1 P1-1 (background commit), P1-2 (R0 false), P1-3 (SDK abort wrapping), and P2-2 (lifecycle ordering). Annotations show which finding each block addresses.

```ts
// session-reset.ts, near top of file:
const RESET_OPERATION_TIMEOUT_MS = 30_000;

// helper — module-private (round-0 decision, kept in round-1):
function classifyAbort(reason: unknown): Error {
  if (reason instanceof TimeoutReason) return new ResetCancelledError('timeout');
  if (reason instanceof ResetCancelledError) return reason; // user-fired with explicit reason
  return new ResetCancelledError('user');
}

// inside reset(), after all existing guards (line 78–114):
handler.setResetInFlight(threadId, true);
try {
  const operationCtrl = new AbortController();
  const merged = mergeSignals(options?.signal, operationCtrl.signal)!;
  // merged is non-null because operationCtrl.signal is always present.

  // [P1-2 fix] Pre-check BEFORE constructing summaryPromise.
  // Async function bodies run synchronously up to the first await, so a
  // summaryPromise constructed before this check would call provider.chat
  // (line 172 of agent-worker.ts) before the race could reject.
  if (merged.aborted) {
    throw classifyAbort(merged.reason);
  }

  handler.setCurrentResetController(threadId, operationCtrl);
  let timer: NodeJS.Timeout | undefined;
  try {
    // summaryPromise body — same as existing lines 118–240, except:
    //   (a) respondDeterministic forwards `merged` (signal forwarding)
    //   (b) [P1-1 fix] post-await abort gate before validation/persist
    const summaryPromise = (async () => {
      const messages = handler.getCurrentSegmentMessages(threadId);
      const existing = effectiveResetSnapshots(threadId, this.db, this.artifactDb);
      // ... build priorSummariesMsg, summaryMsg, facilitatorMessages
      //     (same as current lines 118–160)

      const response = await this.facilitator.respondDeterministic(
        facilitatorMessages,
        'synthesizer',
        merged,                    // ← signal forwarding (§3.1)
      );

      // [P1-1 fix] If cancel/timeout landed while LLM was in flight,
      // refuse to commit even if the SDK happened to return a complete
      // response before honoring the abort. Without this gate, a slow-to-
      // honor SDK can let the cancelled reset commit DB+segment state
      // AFTER cleanup ran — a background-write bug. Codex round-1 P1-1.
      if (merged.aborted) {
        throw classifyAbort(merged.reason);
      }

      const summaryMarkdown = response.content;
      const validation = validateResetSummaryMarkdown(summaryMarkdown);
      if (!validation.valid) {
        throw new MalformedResetSummaryError(validation.missingSections);
      }
      // ... parseSummaryMetadata, computeNextSegmentIndex, build snapshot
      //     (same as current lines 183–206)

      this.db.recordSnapshot(snapshot);

      // Synchronous block 208–233 (codex round-1 verified): no await between
      // recordSnapshot and openNewSegment, so race rejection cannot preempt.
      // Existing seal/open rollback via safeDeleteSnapshot is unchanged.
      try {
        handler.sealCurrentSegment(threadId, snapshotId);
      } catch (sealErr) {
        this.safeDeleteSnapshot(snapshotId, sealErr);
        throw sealErr;
      }
      try {
        handler.openNewSegment(threadId);
      } catch (openErr) {
        try { handler.unsealCurrentSegment(threadId); } catch {}
        this.safeDeleteSnapshot(snapshotId, openErr);
        throw openErr;
      }

      return { snapshotId, summaryMarkdown, metadata: snapshot.metadata, segmentIndex };
    })();

    const racePromise = new Promise<never>((_, reject) => {
      timer = setTimeout(() => {
        // Single timer drives BOTH abort and reject (v0.5.3 §5.1 site 5).
        // Reject synchronously before abort so race winner is deterministic.
        reject(new ResetCancelledError('timeout'));
        operationCtrl.abort(new TimeoutReason(RESET_OPERATION_TIMEOUT_MS));
      }, RESET_OPERATION_TIMEOUT_MS);

      // External cancel path: user-fired controller.abort() must also reject.
      // (merged.aborted at function entry is handled by the §4.1 P1-2 gate
      // above — by here, merged is not yet aborted.)
      merged.addEventListener('abort', () => {
        reject(classifyAbort(merged.reason));
      }, { once: true });
    });

    // [P1-3 fix] Wrap race-await in try/catch and reclassify SDK aborts.
    // If a provider SDK rejects summaryPromise with APIUserAbortError /
    // AbortError BEFORE racePromise rejects (SDK abort listener fires
    // faster than our addEventListener), the caller would see a raw SDK
    // error class instead of ResetCancelledError. Reclassify via merged.reason
    // — same pattern as artifact-invoke.ts:48 (codex round-1 P1-3).
    try {
      return await Promise.race([summaryPromise, racePromise]);
    } catch (err) {
      if (isAbortError(err) && merged.aborted) {
        throw classifyAbort(merged.reason);
      }
      throw err;
    }
  } finally {
    // Inner cleanup: clear timer FIRST (so any late-firing setTimeout
    // callback doesn't run after controller is null), then clear controller.
    if (timer) clearTimeout(timer);
    handler.setCurrentResetController(threadId, null);
  }
} finally {
  // Outer cleanup: clear flag last so a second /councilcancel arriving
  // during cleanup never sees `resetInFlight=false` AND a non-null
  // controller simultaneously (round-1 P2-2 ordering).
  handler.setResetInFlight(threadId, false);
}
```

**Lifecycle ordering invariant (round-1 P2-2)**:
- During reset: `resetInFlight=true` AND `currentResetController=<controller>`
- After cleanup: `resetInFlight=false` AND `currentResetController=null`
- Cleanup transition: controller cleared FIRST (inner finally), flag cleared LAST (outer finally). A second `/councilcancel` reading `resetInFlight=true` is guaranteed to see a non-null controller; reading `resetInFlight=false` is guaranteed to see `null` controller.

**Why two nested try/finally instead of one**: round-0 used a single outer finally; round-1 codex P2-2 noted that `setCurrentResetController(controller)` placed BEFORE the try would leak `resetInFlight=true` if `mergeSignals` threw at setup. Two finallys make setup-throw rollback automatic: the P1-2 early-throw, the `mergeSignals` setup throw, and any other setup error all go through outer finally and clear the flag.

## 5. Cancel lifecycle state machine

```
            ┌─────────┐
            │  idle   │
            └────┬────┘
                 │ user fires /councilreset
                 │ guards pass
                 │ resetInFlight = true
                 │ currentResetController = new AbortController()
                 ▼
         ┌─────────────────┐
         │   in-flight     │
         │ - LLM call      │
         │ - DB recordSnap │
         │ - seal segment  │
         │ - open segment  │
         └─┬───────┬───────┘
           │       │
   summary │       │ abort fires
   resolves│       │ (timeout OR /councilcancel)
   AND seal│       │
   AND open│       │
   succeed │       │
           │       │
           ▼       ▼
       settled   cancelled-or-timed-out
       (return)   (throw ResetCancelledError)
                  ↓
                  R1/R2/R3/R4 rollback per §6
                  ↓
       finally: clearTimeout, controller=null, resetInFlight=false
```

## 6. Race scenarios

**Load-bearing JS guarantee** (codex round-1 should verify): `Promise.race([summaryPromise, racePromise])` cannot inject the race-reject between sync statements inside `summaryPromise`'s body. Once `summaryPromise`'s async function resumes from an `await` and enters a synchronous block, that block runs to completion (or the next `await`) before any other microtask — including the race-reject — is dispatched. This is what makes R2/R3 below collapse into R4.

**The R2/R3 collapse is load-bearing** (codex round-1 verified): cancel cannot arrive between line 208 and line 233 because that block is synchronous (no `await`), AND `Promise.race` cannot preempt sync code inside summaryPromise's body. This means we DO NOT need cancel-aware rollback inside seal/open. The existing `safeDeleteSnapshot` (line 249) covers seal-fail / open-fail, and cancel-after-LLM is exclusively R1 (LLM in-flight, no commit), R1' (LLM done but cancel landed first, post-await gate refuses to commit), or R4 (synchronous block already ran, success).

**R1' is the round-1 P1-1 critical fix**: without the post-await abort gate, the SDK could return a complete response after cancel landed, and summaryPromise would proceed to write DB / mutate segment in the background AFTER outer cleanup ran. The gate enforces the cancellation invariant (§2.2): cancel either prevents the snapshot row from being written, or arrives too late and reset commits. Cancel never partially commits.
